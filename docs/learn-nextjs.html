Chapter 00

How to read this handbook

You already know JavaScript and React. This guide takes you from there to fluent in this codebase — teaching each Next.js idea, then showing exactly how My Piano Diary uses it.

Who this is for

This handbook assumes you're comfortable with React — components, props, state, hooks, and fetching data in a Single-Page App (SPA). It assumes you have never used Next.js, tRPC, Prisma, or NextAuth. Every concept is taught from first principles, then grounded in real code from this repository so you learn the tools and the app at the same time.

The app is a real product: a studio manager for a piano teacher. Wherever a concept could feel abstract, we anchor it to something concrete — a lesson, a payment, a monthly report.

The two mental shifts that matter most

Almost everything that feels new comes down to two ideas. Hold onto these and the rest falls into place:

1 · Rendering moves to the server

In your SPA, every component ran in the browser and fetched data with useEffect. In Next.js, components run on the server by default — they can await the database directly and send finished HTML. You opt back into the browser with one line: "use client".

2 · The API is just typed functions

There are no REST endpoints to design or fetch() calls to write. With tRPC you call a server function like api.payment.addTransaction and TypeScript checks the arguments and the return type across the network — as if it were a local call.

Analogy

If you remember one sentence: the server does the work up front, and one language (TypeScript) flows unbroken from the database to the button in the browser. Chapters 03–06 build this picture; Chapter 05 shows the whole round-trip in one diagram.

Chapter 01

The big picture

Before any code: what does this app do, and what are all these libraries for?

What My Piano Diary is

My Piano Diary is a studio-management app for a piano teacher. One teacher signs in and manages their whole practice:

  • Students — each with an in-person rate and an online rate.
  • Lessons — scheduled on a calendar (one-off or recurring), each marked pending, complete, or cancelled.
  • Payments — each month, what a student owes is computed from their completed lessons, and the teacher records what was actually paid.
  • Reports — a monthly write-up per student.
  • Pieces — the repertoire library, with difficulty ratings.
  • A dashboard — earnings trends, top students, quick insights… and a playful birthday mode.

Most of the genuine complexity lives in the money: rates are frozen onto each lesson so history never re-prices, and everything is bucketed into months using the teacher's timezone. That's Chapter 10 — the payoff.

Chapter 03

Routing with the App Router

How folders become URLs, what each special file does, and the route-group trick that splits this app into a logged-out world and a dashboard world.

If you're coming from plain React (Create React App, Vite + React Router), the mental model you have is: "I write components, and somewhere I hand-wire a <Routes> config that maps URL strings to those components." Next.js throws that config away. In the App Router (the modern Next.js routing system, living in the src/app/ directory), **the folder structure on disk is the routing table.** A folder named students under app/ becomes the /students URL. There is no route array to maintain — you create a folder, drop a file with a special name inside it, and the URL exists.

The second big shift is that files here are React Server Components by default. In your SPA, every component ran in the browser. Here, a page.tsx runs on the server first — it can await a database call directly, with no useEffect, no loading spinner wiring, no API endpoint in between — and Next.js streams the resulting HTML to the browser. You only opt a file into the browser (making it a "Client Component") by writing "use client" at the very top. That one line is the border crossing between the two worlds, and half of learning the App Router is learning which side of that border a given file lives on.

The App Router gives special meaning to a handful of filenames. page.tsx makes a folder publicly routable. layout.tsx wraps every page beneath it in shared UI (and persists across navigations — it doesn't re-render when you move between sibling pages). loading.tsx is shown automatically while a server page is fetching. error.tsx catches crashes in that part of the tree. not-found.tsx renders your 404. route.ts (note: .ts, not .tsx) is not a page at all — it's a raw HTTP endpoint handler, the App Router's version of an Express route. This repo uses every one of these.

Finally, this codebase leans on one clever organizational trick: route groups, written as folders in parentheses like (auth) and (root). The parentheses mean "group these routes together so they can share a layout, but don't add a URL segment." So (root)/dashboard/page.tsx serves the URL /dashboard — the (root) is invisible in the address bar. This project splits its pages into two worlds: (auth) holds the logged-out pages (login, register) with their own bare layout, and (root) holds the authenticated dashboard — every page under it is wrapped in the sidebar shell and gated behind a server-side login check. That's the shape you'll be working inside every day.

This project was generated from the T3 Stack (create-t3-app), which is why you'll see tRPC, Prisma, and NextAuth pre-wired. That's a separate topic, but it explains the api/trpc and api/auth route handlers you'll meet at the end.

Folders become URLs: the filesystem *is* the router

In a React SPA you write something like <Route path="/students" element={<Students/>} />. You maintain that mapping by hand, and it lives in one central file. The App Router deletes that file. Instead, the path of a folder under src/app/ becomes the URL, and a page.tsx inside that folder is what renders at that URL.

So in this repo: - src/app/(root)/students/page.tsx → serves /students - src/app/(root)/lessons/page.tsx → serves /lessons - src/app/(root)/dashboard/page.tsx → serves /dashboard

The rule is simple: a folder is only publicly reachable if it contains a page.tsx (or a route.ts — more on that later). A folder without one of those isn't a route; it's just a place to keep files. This project uses that fact heavily — notice all the _components folders (the leading underscore is a convention, discussed in its own section) that hold UI but never become URLs.

A subtle but important consequence: because there's no central route file, to find a route you navigate the folder tree, and to add a route you make a folder. There's nothing to register. When you're hunting for 'what renders at /payments?', the answer is always src/app/**/payments/page.tsx.

One gotcha to file away now: the default export of page.tsx is the page component, and Next.js cares that it's the default export. A named export won't be picked up as the page.

Analogy

Think of React Router's route config as a phone book you maintain by hand. The App Router replaces it with 'the house's street address is literally where the house physically sits' — the folder path IS the URL.

Gotcha

A folder with no page.tsx (or route.ts) is invisible to the router. That's how _components folders can live right next to pages without ever becoming a URL.

Note

The page must be the DEFAULT export. Next.js looks for export default, not a named export.

Why it's built this way

File-based routing trades a bit of magic ('where did this route come from?') for a huge win: routes are impossible to forget to register, colocation of a page with its helpers is natural, and code-splitting per route happens automatically because each folder is its own boundary.

Server Components by default, and the "use client" border

This is the concept that trips up every React dev arriving at Next.js, so internalize it early. Every file under src/app/ is a React Server Component (RSC) unless it says otherwise. A Server Component runs on the server, never ships its JavaScript to the browser, and — crucially — can be async and await data directly.

Look at (root)/reports/page.tsx: the function is async, and it just awaits the database through tRPC as if it were local code. There's no useEffect, no useState, no loading flag, no fetch to an API route. The data is fetched on the server before any HTML is sent:

export default async function Reports(props) {
  const students = await api.student.getAll();
  // ...render with the data already in hand
}

Compare that to your SPA muscle memory, where you'd render an empty shell, fire a useEffect, hit an endpoint, store the result in state, and re-render. The Server Component collapses all of that into a straight-line await.

The catch: Server Components cannot use hooks (useState, useEffect), event handlers (onClick), or browser APIs — because none of that exists on the server. The moment you need interactivity, you cross the border by putting "use client" as the literal first line of the file. That makes it a Client Component: it behaves like the React you already know (runs in the browser, has hooks and events), but loses the ability to be async/await data server-side.

You can see both kinds side by side in this repo. (root)/reports/page.tsx is a Server Component (async, awaits data, no directive). (root)/_components/site-header.tsx opens with "use client" because it uses useState, useEffect, useSession, and an onClick. The common pattern here — and it's a good pattern — is: a thin async Server Component page fetches data, then passes it as props into a "use client" component that handles all the interactivity. reports/page.tsx doing <ReportsPage students={...} initialReports={...} /> is exactly that handoff.

Why

Server Components let a page await the database with zero client JS and zero fetch boilerplate. The data is baked into the HTML before it reaches the browser, so there's no spinner-then-pop-in flash and no extra round trip.

Gotcha

"use client" must be the VERY FIRST line of the file (above imports). Forget it and any useState/onClick in that file throws a build error like 'You're importing a component that needs useState...'.

Tip

The idiomatic move in this repo: keep page.tsx as a small async Server Component that fetches, and push all interactivity into a _components/*-page.tsx client component it renders. Fetch on the server, interact on the client.

Gotcha

In newer Next.js, params and searchParams handed to a page are Promises — notice reports/page.tsx does await props.searchParams. You must await them.

Why it's built this way

Defaulting to Server Components shrinks the JavaScript bundle (server-only code never ships) and removes the entire 'fetch-in-useEffect' dance for read paths. Interactivity is opt-in precisely where it's needed, so the app stays fast without the developer micromanaging what runs where.

The special files: page, layout, loading, error, not-found

The App Router reserves a small vocabulary of filenames. Each one plugs into the framework at a specific spot. This repo uses all of them, so here's the whole set with the real files:

page.tsx — makes the folder a public route (covered above).

layout.tsx — shared UI that wraps every page beneath it. Its defining superpower is that it persists across navigation: when you move from /students to /lessons, the layout does not unmount and re-render — only the {children} swap out. That's why the sidebar in this app keeps its scroll position and open/closed state as you click around. A layout receives children and must render them.

loading.tsx — an instant fallback UI shown automatically while a Server Component page is awaiting its data. Under the hood Next.js wraps the page in a React <Suspense> boundary and uses your loading.tsx as the fallback. You write zero Suspense code; you just provide the file. This repo's root loading.tsx renders an <AppLoader/> spinner centered on the screen.

error.tsx — a React error boundary for that segment. If a page (or anything it renders) throws, this component renders instead of a white screen. It must be a Client Component ("use client"), and it receives two props: the error object (with an optional digest id) and a reset() function that retries rendering the segment. This project has three of them at different levels — root, (auth), and (root) — each logging to a monitoring helper and offering a 'Try Again' button wired to reset().

not-found.tsx — renders when a route doesn't exist (a 404), or when code calls notFound(). The root one here logs the 404 and offers 'Go Back' / 'Go Home'.

route.ts — a raw HTTP endpoint, NOT a page. Covered in its own section below because it's conceptually different.

Why

A layout persists across navigation between its child pages — it doesn't remount. That's the whole reason the sidebar keeps its state as you click between /students, /lessons, /reports.

Gotcha

error.tsx and not-found.tsx that use hooks MUST start with "use client" — error boundaries are inherently a client concept in React. All three error.tsx files here do.

Tip

error.tsx only catches errors from the segments BELOW/beside it, not from its own sibling layout.tsx. To catch a layout's own crash you need an error.tsx one level up. That's part of why this app has error boundaries at multiple levels.

Note

loading.tsx only kicks in for async Server Component pages that actually suspend. A fully client-rendered page won't trigger it.

Why it's built this way

By making these behaviors filename-driven, Next.js turns 'every route needs a loading state and an error boundary' from boilerplate you'd forget into a convention you get for free. Drop a file in the folder and the whole subtree inherits the behavior.

Route groups: (auth) and (root) — folders that don't touch the URL

Here's the organizational backbone of this codebase. A folder whose name is wrapped in parentheses — (auth), (root) — is a route group. The parentheses tell Next.js: 'use this folder to organize files and to apply a shared layout, but do NOT add a segment to the URL.'

So despite living at src/app/(root)/dashboard/page.tsx, the dashboard's URL is just /dashboard — the (root) is stripped out. Same for (auth)/login/page.tsx → /login. The parentheses folder is purely a grouping device; it's invisible to the router's URL computation.

Why would you want that? Because a layout.tsx applies to everything in its folder subtree. Route groups let you give different sets of pages different layouts without those groupings showing up as URL prefixes. This project splits the app into exactly two worlds:

  • (auth) — the logged-out pages: /login and /register. These need a clean, full-bleed layout with no sidebar (you're not authenticated yet, so there's no dashboard chrome to show). In this repo (auth) has its own error.tsx but no dedicated layout.tsx, so these pages fall back to the root layout only.
  • (root) — the logged-in dashboard: /dashboard, /students, /lessons, /reports, /payments, /pieces, /calendar, /profile, /notifications, /updates. Every one of these is wrapped by (root)/layout.tsx, which renders the sidebar shell AND enforces the login check.

Without route groups, to give the dashboard pages a shared layout you'd have to nest them all under a real /dashboard/... URL prefix, polluting every address. Route groups give you the shared layout while keeping URLs flat and clean. That's the tradeoff they solve.

Note also the pages that live directly under src/app/ with no group — /birthday-game, /birthday-room, /forever. These deliberately sit outside both groups, so they get neither the auth gate nor the sidebar — they're standalone full-screen experiences.

Analogy

A route group is like a folder on your computer you make just to stay organized — it groups files, but it isn't part of any file's public 'address' you'd hand to someone else.

Why

The (auth) vs (root) split exists so logged-out pages get a bare layout and logged-in pages get the sidebar shell + auth gate — two completely different layouts, but both keep flat URLs like /login and /dashboard.

Gotcha

You can't have two route groups both define a page at the same resolved URL — e.g. (auth)/dashboard and (root)/dashboard would both resolve to /dashboard and collide. Group names are invisible, so they can't disambiguate URLs.

Note

birthday-game, birthday-room, and forever live at the top of app/ outside any group, so they intentionally skip the (root) auth gate and sidebar.

Why it's built this way

Route groups let the project express 'these pages share chrome and rules' as a folder boundary, without leaking that boundary into the URL. It keeps auth-vs-app a physical, visible split in the tree while URLs stay short and user-friendly.

Nested layouts: the root shell and the (root) dashboard shell

Layouts compose by nesting. When you visit /dashboard, Next.js renders every layout.tsx from the app root down to the page, each wrapping the next. For this repo the chain is: app/layout.tsx (root) → app/(root)/layout.tsx (dashboard shell) → app/(root)/dashboard/page.tsx. The root layout's {children} is the dashboard layout, whose {children} is the page.

The root layout (app/layout.tsx) is special: it's the only layout that renders the <html> and <body> tags, and it's required. Several things worth understanding are happening here: - Fonts via next/font/google. Instead of a <link> to Google Fonts (an external request that hurts performance and privacy), Plus_Jakarta_Sans({...}) downloads and self-hosts the font at build time and hands you a CSS-variable class (--font-plus-jakarta-sans) to apply on <body>. Zero network round-trip to Google at runtime. - Exported metadata. Instead of a <Head> component, App Router pages/layouts export const metadata and Next.js turns it into <title>/<meta> tags. Here it pulls title and description from APP_CONFIG. - <Providers> wraps all children — this is where the client-side context providers live (SessionProvider, tRPC, toaster, etc.), isolated into one "use client" component so the layout itself can stay a Server Component. - suppressHydrationWarning on <html> — see the gotcha below.

The (root) layout (app/(root)/layout.tsx) is where the authenticated experience is assembled. It does two jobs. First, it's the auth gate: it awaits the session on the server and, if there's none, calls redirect("/login") before any dashboard UI is ever sent to the browser. Because this runs server-side in a layout, every page under (root) is protected in one place — you never repeat the check per page. Second, it renders the sidebar shell: <SidebarProvider> → <AppSidebar/> + <SiteHeader/> + {children}. Since layouts persist across navigation, that sidebar and header mount once and stay put while only {children} swaps as you move between dashboard pages.

Why

Putting the auth check in the (root) LAYOUT means every dashboard page is protected by one server-side redirect. You can't forget to guard a new page — dropping it under (root) auto-inherits the gate.

Why

next/font downloads and self-hosts the font at build time, so there's no runtime request to Google's servers — faster first paint and no third-party font dependency.

Gotcha

suppressHydrationWarning on <html> silences the harmless mismatch warning that fires when a theme/class is set on the html element before React hydrates. It's scoped to that one element, not a blanket 'ignore all hydration bugs' switch — don't sprinkle it elsewhere.

Tip

Providers is its own "use client" component so the root layout can stay a Server Component. This is the standard pattern: keep layouts server-side, quarantine all client context into one <Providers> child.

Gotcha

redirect() works by throwing a special signal, so any code you write after it won't run. Note app/page.tsx even has unreachable JSX below its redirect('/dashboard') — the redirect short-circuits it.

Why it's built this way

Nesting layouts lets cross-cutting concerns live at exactly the altitude they apply to: html/fonts/providers at the root (everything needs them), the auth gate + sidebar at (root) (only the dashboard needs them). Each concern is written once, at one level, and everything below inherits it.

Dynamic and catch-all segments: [studentId] and [...nextauth]

Static folders match one exact URL. But you often need parameterized routes — 'a reports page for any student id'. The App Router expresses these with bracket folder names:

[studentId] — a dynamic segment. The folder (root)/reports/[studentId]/ matches /reports/anything, and whatever's in that spot is handed to the page as params.studentId. So /reports/abc123 renders this page with studentId === "abc123". In current Next.js, params arrives as a Promise, which is why this page does await props.params. This page also reads the query string (?month=&year=) via the separate searchParams prop — dynamic segments are part of the path, searchParams are the ?key=value part, and they're delivered as two different props.

[...nextauth] — a catch-all segment (the ... spread prefix). api/auth/[...nextauth]/route.ts matches /api/auth/anything/here/deeply/nested — every path under /api/auth/. NextAuth needs this because it exposes many sub-routes under one prefix (/api/auth/signin, /api/auth/callback/google, /api/auth/session, and so on); the catch-all lets a single handler own the whole family of URLs.

So three bracket flavors to recognize: [x] matches exactly one segment, [...x] matches one-or-more segments (a catch-all), and [[...x]] (double brackets, not used here) would match zero-or-more (optional catch-all).

Note

[folder] = one segment. [...folder] = catch-all (one or more segments). [[...folder]] = optional catch-all (zero or more). This repo uses the first two.

Gotcha

params and searchParams are Promises in current Next.js — you MUST await them. Reading props.params.studentId without awaiting gives you a Promise, not the string.

Tip

Path params ([studentId]) come from the folder name; query params (?month=1) come from searchParams. They're separate props for a reason — don't look for ?month= inside params.

Why it's built this way

Bracket-folder routing keeps parameterized URLs in the same filesystem-as-router model as everything else — no separate pattern-matching config. The catch-all is exactly the escape hatch a library like NextAuth needs to own an entire URL namespace with one file.

route.ts vs page.tsx: API route handlers

Everything so far rendered UI. But the app/ directory can also host raw HTTP endpoints, and that's what a route.ts file is. The distinction is sharp and worth stating plainly:

  • page.tsx returns JSX (a React component). Visiting it renders a web page.
  • route.ts exports functions named after HTTP verbs — GET, POST, etc. — that receive a Request and return a Response. Visiting it runs backend code and returns data (JSON, a redirect, whatever), not a rendered page. It's the App Router's equivalent of an Express/Fastify route handler.

A folder may contain a page.tsx OR a route.ts, never both — they'd both try to own the same URL. That's why API endpoints live under app/api/..., physically separated from the UI pages.

This repo has exactly two route handlers, both from the T3 setup:

api/trpc/[trpc]/route.ts is the single HTTP entry point for the entire tRPC API. Every tRPC call your client makes (api.student.getAll, api.report.getAll, ...) is an HTTP request to /api/trpc/..., and this one handler routes it into appRouter. Notice the last line: export { handler as GET, handler as POST } — one function serving both verbs, because tRPC uses GET for queries and POST for mutations. It also wires up error logging via onError.

api/auth/[...nextauth]/route.ts is even smaller — it just re-exports the GET and POST handlers that NextAuth generates. NextAuth owns all the sign-in/callback/session plumbing; this file's only job is to mount those handlers at the /api/auth/* catch-all so the browser (and OAuth providers redirecting back) can reach them.

Note

route.ts is a .ts file, not .tsx — it never returns JSX. It exports functions named GET/POST/PUT/PATCH/DELETE that take a Request and return a Response.

Gotcha

You cannot put both page.tsx and route.ts in the same folder — they'd collide on the same URL. UI and API endpoints therefore live in separate folders (hence the app/api/ tree).

Why

tRPC funnels its entire API through one [trpc] catch-all route handler, so you write typed procedures in server/api/ instead of hand-authoring dozens of REST endpoints. This one route.ts is the bridge between HTTP and that typed layer.

Tip

When you see api.student.getAll() in a client component, remember it becomes an HTTP request to this /api/trpc/[trpc] handler under the hood — that's the seam between the tRPC magic and plain HTTP.

Why it's built this way

Separating route.ts (data) from page.tsx (UI) keeps a clean line between 'render a page' and 'answer an HTTP request', while still living in the same file-based routing system. T3 leverages it to expose exactly two endpoints — tRPC and auth — instead of a sprawl of REST routes.

The _components convention and the full route tree

One last convention before the map: folders and files prefixed with an underscore, like (root)/dashboard/_components/, are private folders. The underscore tells Next.js 'never treat anything in here as a route, even if it somehow contains a page-like file'. This project uses _components everywhere to colocate a route's UI pieces right next to the page.tsx that uses them — the dashboard's cards live in dashboard/_components/, the reports UI in reports/_components/, and so on. It's the App Router blessing of 'keep a feature's parts together' without those parts leaking into the URL space.

(You'll even find a leftover (root)/students/[id]/reports/_components/report-view.tsx that has no page.tsx beside it — so /students/[id]/reports is NOT a live route; that component is currently just an orphaned/unused file. A good reminder that only a page.tsx or route.ts creates a URL.)

Chapter 04

Server & Client Components

The single biggest mental shift from a React SPA: components render on the server by default, and one directive — "use client" — is the border into the browser.

If you come from plain React or a Vite/CRA SPA, your mental model is: "everything is a component, everything runs in the browser, and I fetch data with useEffect + fetch after the page loads." The single biggest shift in the Next.js App Router is that this is no longer true. By default, every component in the app/ directory is a React Server Component (RSC) — it runs on the server, never in the browser, and ships zero JavaScript to the client. Only the HTML it produces is sent down. A component becomes interactive (browser-side) only when you explicitly opt in with the "use client" directive at the top of the file.

Why would anyone want components that don't run in the browser? Because a huge amount of UI is just "read data, render markup" — a dashboard layout, a page that lists lessons, a report. If that work happens on the server, you can await a database query directly inside the component, you can read secrets and cookies, and you send the user finished HTML with none of the fetching logic (or the libraries that power it) shipped to their phone. The parts that genuinely need to be interactive — a form with validation, a dialog that opens on click, a table with client-side filtering — are the minority, and those are the ones you mark "use client".

This repo is a textbook example of the split. Look at the numbers: exactly one file in src/server/actions/ carries "use server", while 84 files carry "use client". The server-only surface is small and deliberate. The page.tsx and layout.tsx files are (mostly) async Server Components that authenticate the user and fetch data, then hand that data as props to "use client" components living in _components/ folders that do the interactive work — forms, dialogs, animated cards, filterable tables. The boundary between the two worlds is a real line in the code, and props that cross it must be serializable (plain data, not functions or class instances).

Two more pieces complete the picture. Server Actions ("use server") let a client component call a server function as if it were a local async function — no manual API route, no fetch boilerplate. This repo uses them for login and registration. And data fetching in Server Components doesn't go over the network at all: an RSC calls the tRPC caller directly (@/trpc/server), invoking your API procedures as in-process function calls, while client components call the same procedures over HTTP through a React Query client (@/trpc/react). Same API, two transports, chosen automatically by which side of the boundary you're on.

This document teaches each concept from first principles and then shows the exact file in this repo that demonstrates it.

Server Components are the default — and they can be async

In the App Router, any .tsx file under src/app/ that does not start with "use client" is a Server Component. It executes on the server (during the request, or at build time), and the browser only ever receives the HTML it rendered — never the component's own JavaScript. This has three superpowers a plain-React component can never have:

  1. It can be async. You can await right in the component body. There is no useEffect, no loading state, no useQuery — you just await the data and render it.
  2. It can touch the server directly — the database, environment secrets, request headers, cookies — because that code never leaves the server.
  3. It ships no JS for itself. Less JavaScript to download and execute means a faster page.

Look at this repo's protected layout. It's an async function, it awaits the auth session on the server, and if there's no session it redirects before any UI is sent. A plain-React SPA can't do this — it would have to render something, then check auth in an effect, then bounce the user, flashing protected chrome in between. Here the redirect happens on the server before a single byte of the dashboard reaches the browser.

The lessons/page.tsx file shows the data-fetching half: it's async, and it simply awaits two API calls, then passes the results down as props. No useEffect, no spinner logic in the page itself.

The cost of these powers: a Server Component cannot use useState, useEffect, or any hook; cannot use onClick/onChange or any event handler; and cannot touch browser-only globals like window or localStorage. None of those things exist on the server. The moment you need one, you cross into client territory (next section).

Browser request
     │
     ▼
[ Server Component ]  ← runs on server, can await db/auth/secrets
   await api.lesson.getAll()      ships NO JS for itself
     │  renders HTML + serialized props
     ▼

Analogy

Think of a Server Component like a PHP or Rails template that can also await data — it runs once on the server to produce HTML. Think of a Client Component like a classic React component that boots up and lives in the browser. Same JSX syntax, two completely different runtimes.

Gotcha

There is no import or config that makes a file a Server Component — it's the absence of "use client" at the top. If you paste a useState into a page.tsx and it errors with "You're importing a component that needs useState... mark it with use client", that's the boundary talking.

Why

Why default to server? Because most UI is read-and-render. Making that the default means the fast, secure, zero-JS path is what you get for free, and you only pay for interactivity where you actually opt into it.

Why it's built this way

This repo consistently keeps layout.tsx/page.tsx as server-side (auth + fetch) and pushes interactivity into _components/. That keeps secrets and DB access on the server, does auth redirects before any protected UI leaks, and gives client components ready-made initial data instead of a load-then-fetch waterfall.

The "use client" directive: opting into the browser

When a component needs to do something in the browser — hold state, respond to clicks, run an effect, read localStorage, use a ref — you put "use client" as the very first line of the file. This marks the file (and everything it imports that isn't already server-only) as a Client Component: its JavaScript is bundled and shipped to the browser, where React hydrates it into a live, interactive component exactly like in a normal SPA.

The "use client" directive is a boundary marker, not a per-component switch. Once a module is "use client", every component it imports is pulled into the client bundle too (unless that import is itself a server-only thing passed through as a child). So you typically place the directive at the top of the leaf files that need interactivity.

In this repo, "use client" shows up in 84 files, and they fall into clear buckets: - Forms — login-form.tsx, register-form.tsx, student-form.tsx, piece-form.tsx, profile-form.tsx (they use react-hook-form, useState, onSubmit). - Dialogs / sheets / interactive overlays — lesson-dialog.tsx, attendance-dialog.tsx, payment-history-dialog.tsx, student-sheet.tsx. - Interactive pages — lessons-page.tsx, payments-page.tsx, reports-page.tsx (filtering, sorting, live tRPC queries). - Design-system primitives — nearly everything in src/components/ui/ (dialog.tsx, select.tsx, popover.tsx, tabs.tsx…) because Radix UI is built on hooks and refs. - Providers — src/components/providers/index.tsx, src/trpc/react.tsx.

The login form is a clean example: it declares "use client", then uses useRouter, useState, react-hook-form, and an onSubmit event handler — none of which can exist on the server.

Tip

Rule of thumb for this codebase: if the file lives in a _components/ folder or in src/components/ui/, it's almost certainly "use client". If it's a bare page.tsx/layout.tsx, it's almost certainly a Server Component.

Gotcha

"use client" does NOT mean "only runs in the browser." Client Components still render once on the server for the initial HTML (SSR), then hydrate in the browser. That's why code that touches window must be guarded (e.g. typeof window === "undefined"), as src/trpc/react.tsx does.

Why

The directive is per-file rather than per-component so the bundler can statically split server-only and client code without analyzing every JSX node. It's a coarse but predictable boundary.

Why it's built this way

Concentrating "use client" in leaf _components/ and ui/ files keeps the interactive JavaScript bundle scoped to what genuinely needs it, while the pages/layouts that orchestrate them stay on the server. It's the standard T3/App-Router pattern: server shells, client leaves.

The boundary: how props cross from server to client

The server/client split creates a literal seam in your component tree. A Server Component can render a Client Component and pass it props — that's exactly what lessons/page.tsx does when it renders <LessonsPage initialLessons={lessons} ... />. But there's a hard rule at this seam: everything that crosses from server to client must be serializable.

Why? Because the server renders, then must package the props up, embed them in the HTML payload, ship them to the browser, and React re-creates them there to hydrate the Client Component. You can't serialize a running function or a live database connection over the wire. So the following can cross: strings, numbers, booleans, null, plain objects and arrays of those, and — thanks to React's RSC serialization — Date, Map, Set, and BigInt. The following cannot cross: functions, class instances with methods, Symbols, JSX event handlers.

This is the #1 newcomer trip-up: you try to pass a callback like onSave={handleSave} from a Server Component into a Client Component and get a runtime error. The fix is to move the callback's definition into the client side (or use a Server Action, which is a special serializable reference — see the next section).

In this repo the pattern is: the async page fetches domain data (Prisma models come back as plain objects with Date fields), maps/trims it if needed, and passes it as initialLessons / students props into the "use client" LessonsPage. Those are all plain data, so they serialize cleanly. The client component then takes over — it seeds its interactive state from those initial props and can re-fetch live data via tRPC hooks from there.

Gotcha

You cannot pass an event handler or any function as a prop from a Server Component to a Client Component. If you need the child to call back into the server, use a Server Action (a serializable reference) instead of a plain function.

Note

Date objects survive the server→client boundary in RSC — you don't have to stringify them. Prisma returns Date fields, and they arrive intact as real Date instances on the client.

Analogy

Think of the boundary like JSON.stringify on steroids: whatever you pass has to be reconstructable from data alone. A recipe (data) crosses fine; a chef mid-cooking (a live function/closure) does not.

Note

Precise mechanic: the server page doesn't seed React useState — it passes its fetched data as React Query initialData to api.lesson.getAll.useQuery(...), and only when the filters are at their defaults (initialData: isDefaultFilters ? initialLessons : undefined). So the client useQuery mounts already-populated, then refetches live as the user filters. That's the real server→client handoff.

Why it's built this way

The page deliberately trims students down to { id, name } before passing them across — sending only the fields the client needs keeps the serialized HTML payload small and avoids leaking unrelated student data (rates, contact info) into the browser.

Server Actions: calling the server without an API route

A Server Action is a function that runs on the server but that you can call directly from a Client Component as if it were a local async function. You create one by putting "use server" at the top of a file (making every export a Server Action) or at the top of a function body. Under the hood, Next.js turns that function into a secure RPC endpoint; when the client 'calls' it, the framework serializes the arguments, POSTs them to the server, runs the real function there, and returns the serialized result. You write zero fetch, zero route handler, zero JSON plumbing.

This repo's only "use server" file is src/server/actions/auth-actions.ts. The directive on line 1 marks both loginAction and registerAction as Server Actions. Walk through loginAction: - It receives a FormData (a serializable type that's ideal for Server Actions). - On the server it validates with Zod, calls NextAuth's signIn with the credentials, then queries the database (db.user.findUnique) for the user record. - It returns a plain object { success, message, user } — serializable, so it can travel back to the client. - Errors are caught and turned into { success: false, message } rather than thrown, so the client always gets a clean, serializable result.

On the client, login-form.tsx imports loginAction and calls it like any async function inside onSubmit: it builds a FormData, awaits loginAction(formData), and branches on result.success. Notice what's happening — a browser component invoked server-only code (bcrypt hashing lives in the sibling registerAction, DB access, NextAuth) with a plain function call. The bcryptjs, Prisma, and NextAuth server internals are never shipped to the browser; only the thin call reference is.

After a successful login the client updates a Zustand store, router.push("/dashboard"), and crucially router.refresh() — which tells the App Router to re-run the Server Components for the new route so the freshly-authenticated session is reflected.

login-form.tsx ("use client")
   onSubmit ──await loginAction(formData)──►  [network POST, handled by Next]
                                                     │
                                                     ▼
                              auth-actions.ts ("use server")
                                bcrypt / Prisma / NextAuth.signIn
                                     │ returns { success, message, user }
   result ◄──────────────────────────┘
   → setUser(); router.push("/dashboard"); router.refresh()
Why

Why a Server Action here instead of tRPC (which this repo uses everywhere else)? Auth needs to set the NextAuth session cookie via signIn, which is cleanest as a server-run action tied to a form submit. Server Actions are the framework-native way to run privileged, cookie-setting logic from a form.

Gotcha

Both the arguments to and the return value from a Server Action must be serializable, exactly like the server→client prop boundary. That's why this action returns a plain result object and swallows errors into { success, message } instead of throwing — a thrown server error would surface to the client as an opaque, non-actionable message.

Tip

router.refresh() after the action is the key to consistency: it re-runs the Server Components so the now-authenticated getServerAuthSession() in the layout sees the new session. Without it, server-rendered UI could still reflect the logged-out state.

Why it's built this way

Keeping the entire auth surface in one small "use server" file (the only one in the repo) means bcrypt, the Prisma client, and NextAuth internals stay strictly server-side, while the client form stays a thin, dumb caller. It's the minimal secure surface for the one flow that must set cookies.

Two ways to call tRPC: server caller (no network) vs client hooks (over HTTP)

This repo exposes the same tRPC API through two different entry points, and which one you use depends on which side of the boundary you're on.

In a Server Component, you import api from @/trpc/server. This is a direct in-process caller — createCaller builds an object whose methods invoke your tRPC procedures as plain function calls. There is no HTTP request; the file is even marked import "server-only" so it can never be bundled into the browser. This is why lessons/page.tsx can just await api.lesson.getAll({ from, to }) — it's literally calling the procedure function on the server, sharing the same request context (headers, auth) via the cached createContext. Zero network latency, zero serialization on the way in.

In a Client Component, you import api from @/trpc/react. This is a createTRPCReact client that gives you React Query hooks (api.lesson.getAll.useQuery(...)). These do go over the network: the httpBatchStreamLink sends the call to /api/trpc, batching multiple calls into one HTTP request and using SuperJSON to serialize rich types (Dates, etc.). section-cards.tsx is a good example — a "use client" component that pulls live data with tRPC hooks and animates it.

So the golden rule: Server Components read via @/trpc/server (function call); Client Components read via @/trpc/react (React Query over HTTP). They share one router definition (AppRouter), so types line up end to end — note how the client derives type Lesson = RouterOutputs["lesson"]["getAll"][number] from the same router the server calls.

There's also a HydrateClient export from @/trpc/server (visible in the T3 boilerplate src/app/page.tsx). It's the mechanism for prefetching a query on the server and handing the dehydrated React Query cache to the client so a client hook mounts already-populated. This repo mostly prefers the simpler explicit pattern (fetch in the page, pass as initial* props) over HydrateClient.

Server Component ─ api (@/trpc/server) ─► createCaller ─► procedure()   [in-process, no network]

Client Component ─ api (@/trpc/react) ─► useQuery ─► httpBatchStreamLink ─► POST /api/trpc ─► procedure()
Why

Why call tRPC directly on the server instead of fetching your own /api/trpc? Because an RSC already runs on the server — making an HTTP round-trip to your own backend would be pure overhead. The direct caller shares the request context and skips serialization entirely.

Gotcha

Don't import @/trpc/server into a "use client" file — the import "server-only" guard will throw at build time. In client components you must use @/trpc/react's hooks. The two api objects look similar but are not interchangeable.

Note

Both paths flow through the same AppRouter, so RouterOutputs/RouterInputs give you one source of truth for types across server and client. That end-to-end type link is the whole point of the T3 stack.

Gotcha

The HydrateClient example lives in src/app/page.tsx, but that file calls redirect("/dashboard") before its <HydrateClient> JSX is ever returned — so that block is unreachable leftover create-t3-app boilerplate. HydrateClient is effectively unused in this repo.

Streaming & Suspense: the loading.tsx convention

Because Server Components can be async and take time to fetch, the App Router needs a way to show something while a route segment is still resolving on the server. That mechanism is streaming with Suspense, and the App Router gives you a zero-config on-ramp: drop a loading.tsx file into a route segment, and Next.js automatically wraps that segment's page in a React <Suspense> boundary whose fallback is your loading.tsx.

Concretely: when you navigate to a route whose async Server Component is awaiting data, Next.js instantly streams the surrounding shell (the layout) plus your loading.tsx fallback to the browser, then streams in the real content when the awaits resolve and swaps it in. The user sees an immediate loading state instead of a blank stall — and you never wrote a <Suspense> tag or an isLoading boolean yourself.

This repo has one such file at the root: src/app/loading.tsx. It renders a centered <AppLoader />. Because it sits at src/app/, it's the fallback for the whole app's top-level segment while server work is pending. (There's no per-route loading.tsx in the sub-segments — a search for Suspense/HydrateClient shows the app leans on this single root loader plus client-side RefreshOverlay/AppLoader spinners inside the interactive pages for finer-grained states.)

The key mental shift from SPA-land: in a classic React app you own the loading state — you set isLoading = true, fetch, then flip it. Here, the loading state for server data is a file-system convention. The filename loading.tsx is the API.

Note

loading.tsx only covers loading of Server Component data for that segment (the server awaits). Client-side fetches via tRPC hooks manage their own loading UI — that's why the interactive pages here also use AppLoader/RefreshOverlay internally.

Tip

Want a per-page skeleton? Add a loading.tsx inside that route's folder (e.g. src/app/(root)/reports/loading.tsx). The nearest loading.tsx up the tree wins for a given segment.

Analogy

loading.tsx is to a route what a <Suspense fallback> is to a component — except the App Router writes the <Suspense> for you, and the file's location decides which content it wraps.

Why it's built this way

A single root loading.tsx gives every server-rendered navigation an instant, consistent app-wide spinner with almost no code, while the genuinely interactive pages handle their own finer-grained refresh states client-side. It's the minimal-ceremony choice that still avoids blank-screen stalls.

Hydration, providers, and suppressHydrationWarning

Hydration is the moment a Client Component's server-rendered HTML gets 'brought to life' in the browser: React re-runs the component on the client and attaches event listeners, expecting the client's first render to match the server's HTML exactly. If they differ, React logs a hydration mismatch warning and may discard the server HTML for that subtree.

The classic source of mismatch is content that's inherently different on server vs client — the current time, a random value, or a theme read from localStorage/system preference (the server doesn't know the user's theme, the browser does). This repo's root layout sets suppressHydrationWarning on the <html> element for exactly this reason: attributes on <html> are commonly rewritten client-side (by theme logic) before hydration, and this flag tells React 'expect this one element's attributes to differ, don't warn.' It suppresses the warning for that element only — it is not a global mismatch silencer.

Everything interactive is wired up under one "use client" root: src/components/providers/index.tsx (imported as @/components/providers). The root layout.tsx — a Server Component — renders <Providers> around children. Providers is a Client Component that nests the app's context providers: ErrorBoundary, NextAuth's SessionProvider, the tRPC TRPCReactProvider (React Query), a UserStoreProvider (Zustand), plus BirthdayProvider and the Toaster. This is the standard pattern: a server root layout delegates all client context into a single "use client" providers component, so the layout itself stays server-side while context still spans the whole app.

One honest note about theming in this repo: next-themes is imported in exactly one place — src/components/ui/sonner.tsx calls useTheme() to color the toast notifications. But there is no next-themes ThemeProvider wrapping the app. So useTheme() there falls back to its default theme = "system", and toasts follow the OS light/dark preference rather than an in-app theme toggle. If you were expecting a full app-wide theme switcher, it isn't wired here — suppressHydrationWarning is present (a habit from the T3/next-themes template) even though the ThemeProvider that usually accompanies it is not installed.

Chapter 06

tRPC — the typed API

Call server functions from the client as if they were local, with end-to-end TypeScript and zero hand-written REST endpoints.

If you've built React apps before, you know the usual client/server dance: the backend exposes REST endpoints (GET /api/students, POST /api/students), and the frontend calls them with fetch. The problem is that the two halves don't know about each other. The server returns any as far as your React code is concerned. You hand-write TypeScript interfaces for the response, and the moment the backend changes a field, your frontend types quietly lie until something crashes at runtime. Tools like OpenAPI/Swagger or GraphQL fix this with a schema plus a code-generation step — you run a generator, it spits out typed client code, and you re-run it every time the API changes.

tRPC deletes that whole problem by deleting the boundary. Instead of "endpoints," you write plain server functions called procedures, group them into routers, and export a single TypeScript type describing the whole API. The client imports that type (not the code — just the type, which compiles away to nothing) and gives you an object where you can call api.student.getAll.useQuery() and get full autocomplete on inputs, full type inference on outputs, and a red squiggle the instant the server signature changes. No schema file, no codegen step, no REST routes to name. The types flow directly from your database query all the way into your JSX. That's what "end-to-end typesafe" means.

This app is built on the T3 Stack (Next.js + tRPC + Prisma + NextAuth + Tailwind), so the tRPC wiring follows the standard create-t3-app layout. The server half lives in src/server/api/ — trpc.ts is the plumbing (context, procedures, middleware), the routers/ folder holds the actual procedures, and root.ts stitches them into one appRouter and exports the all-important AppRouter type. The client half lives in src/trpc/ — react.tsx wires up the React Query-backed hooks for Client Components, and server.ts gives Server Components a way to call the same procedures directly as async functions. A single Next.js route handler at src/app/api/trpc/[trpc]/route.ts is the actual HTTP endpoint that browser requests hit.

One thing that trips up newcomers: because this is the Next.js App Router, there are two completely different ways to call the same procedure. A React Server Component (RSC) — which runs only on the server — calls await api.student.getAll() directly, like an in-process function call, no HTTP involved. A Client Component ("use client") calls api.student.getAll.useQuery(), a React Query hook that fires an HTTP request to that route handler. Same procedure, two call styles, both fully typed. Most of this guide is about understanding those two paths and the plumbing that makes them safe.

Throughout, the mental model to hold: your procedures are just async functions with a validated input and a typed output. tRPC is the machinery that lets you call them from anywhere — server or browser — as if they were local.

The core idea: procedures, not endpoints

Forget URLs for a moment. In tRPC you write procedures — functions that take a validated input and return a value. A procedure is either a query (a read, like GET) or a mutation (a write, like POST/PUT/DELETE). You group related procedures into a router, and routers can nest. The whole tree of routers is your API surface.

Look at the student router. getAll is a query with no input. getByGuid is a query that takes { id }. create, update, and delete are mutations. Every one is protectedProcedure (more on that later) followed by .query(...) or .mutation(...). The function you pass receives a ctx object (the request context — database handle, session, headers) and, if the procedure declared one, a validated input.

The magic ingredient is .input(zodSchema). You describe the shape of the input with a Zod schema, and tRPC does two things at once: at runtime it parses and validates the incoming data (rejecting bad requests before your code runs), and at compile time it infers the TypeScript type of input from that same schema. One schema, both guarantees. This is why you never see manual request-body casting or req.body as Something — by the time your handler runs, input is fully typed and already validated.

Contrast with REST: there, the request body is unknown/any, you validate it by hand (or don't), and the client has no idea what shape to send. Here the schema is the single source of truth, and — critically — the client will also know the exact input shape because the schema's inferred type rides along in AppRouter.

Analogy

Think of a router as a plain object of async functions, and the procedure as one function on it. api.student.getByGuid is really just studentRouter.getByGuid — tRPC only adds the ability to call it across the network with the types intact.

Why

Query vs mutation isn't just naming. On the client, queries become cache-backed useQuery hooks (auto-dedup, caching, refetch); mutations become useMutation (imperative fire-on-submit with onSuccess/onError). Picking the wrong verb gives you the wrong client ergonomics.

Note

The Zod schemas here live in @/lib/validations/common-schemas (e.g. createStudentSchema, idSchema) and are shared between the router and the front-end forms — one definition guards both ends.

Why it's built this way

Modeling the API as functions instead of URLs is the whole point: a function has a typed signature the compiler can check on both sides, whereas a URL string is opaque. Everything else in tRPC exists to preserve that function-call illusion across the network.

The context: how every procedure gets db + session

Every procedure needs the same ambient things: a database handle and the current user's session. Rather than importing those into each router, tRPC builds a context object once per request and hands it to every procedure as ctx. That's createTRPCContext.

It's deliberately tiny. It calls auth() (NextAuth) to resolve the current session, then returns { db, session, ...opts } where opts carries the request headers. db is the shared Prisma client. So inside any procedure, ctx.db is Prisma, ctx.session is the NextAuth session (possibly null), and ctx.headers is the incoming headers.

The subtle-but-important detail: this same helper is wrapped by two different callers, because there are two ways a request can arrive. When a browser Client Component makes an HTTP call, the route handler wraps createTRPCContext with the real req.headers. When a Server Component calls a procedure directly, src/trpc/server.ts wraps it with the RSC's headers (via next/headers) and tags them x-trpc-source: rsc. Either way, the procedure body sees an identical ctx shape and doesn't care which path it came in on. That uniformity is what lets the exact same procedure serve both an RSC direct-call and an HTTP request.

Analogy

Context is like React context, but for the server request: a bundle of dependencies (db, session) made ambiently available to the whole procedure tree so you don't prop-drill them into every function.

Tip

Want a new per-request dependency (a feature flag, a request-id, a tenant)? Add it here once and it's typed and available as ctx.whatever in every procedure automatically — no per-router plumbing.

Why it's built this way

Centralizing db + session in context means procedures stay pure business logic and testable: you can construct a context with a fake db/session and call the procedure directly. It also guarantees the session is resolved exactly once per request rather than in each procedure.

Initialization: superjson, the Zod error formatter, and the builder pieces

initTRPC.context<typeof createTRPCContext>().create({...}) is where the tRPC instance is born and told about your context type. Two options here matter a lot.

transformer: superjson. Plain JSON can't represent a Date, a Map, a BigInt, or undefined — JSON.stringify(new Date()) gives you a string, and the client would receive a string, not a Date. superjson is a serializer that round-trips those richer types. Because this app's procedures return Prisma rows full of Date objects (lesson dates, createdAt), superjson is what lets lesson.date arrive on the client as an actual Date. The transformer is configured here and must match on the client link (it does — see the React provider). If they ever disagree, you get mangled data.

errorFormatter. When a procedure throws, this shapes the error payload sent to the client. It does two jobs. First, if the error is an INTERNAL_SERVER_ERROR (an unexpected crash, not an expected 401/404/validation), it forwards it to logError for monitoring — expected client errors are deliberately not logged, to avoid noise. Second, and this is the typesafety trick: if the underlying cause is a ZodError (i.e. .input() validation failed), it flattens it into data.zodError. That flattened field-error map ships to the client with types, so a form can highlight exactly which field failed — validation errors become typed data, not opaque strings.

The rest of the file just names the reusable building blocks: createCallerFactory (for server-side callers), createTRPCRouter = t.router (what every router file imports), and — the two you'll use constantly — publicProcedure and protectedProcedure.

Gotcha

The transformer is set in TWO places — here on the server and in the httpBatchStreamLink on the client (src/trpc/react.tsx). Both must be superjson. A mismatch silently corrupts Dates and other rich types.

Why

Only INTERNAL_SERVER_ERROR gets logged for monitoring. Throwing TRPCError({ code: 'UNAUTHORIZED' }) or a NOT_FOUND is an expected outcome, not a bug — logging those would drown real crashes in noise.

Gotcha

The errorFormatter only special-cases INTERNAL_SERVER_ERROR for logging. But this codebase's routers throw plain new Error("Student not found") — and tRPC classifies a plain Error as INTERNAL_SERVER_ERROR, so those do get logged. To mark an error as expected/unlogged you'd throw new TRPCError({ code: "NOT_FOUND" }) instead.

Note

logError in the formatter is a no-op unless NEXT_PUBLIC_ERROR_TRACKING_URL is configured (see env.js). In local dev with no tracking URL set, nothing is actually sent anywhere.

Why it's built this way

Returning validation failures as structured zodError data (instead of a generic 400) is what lets React forms map server-side errors back to specific fields — you get the same field-level error UX whether validation ran on the client or the server, from one Zod schema.

Middleware and the dev-only artificial delay

A tRPC middleware wraps a procedure: it runs code, calls next() to invoke the actual procedure (or the next middleware), and can inspect or modify the result and context. It's the server-side equivalent of an Express middleware, but typed.

This app has one middleware, timingMiddleware, applied to every procedure (both public and protected build on it). It records a start time, calls next(), and — only in development — logs how long the procedure took, e.g. [TRPC] student.getAll took 213ms. Handy for spotting slow queries.

The surprising bit is the artificial delay: in dev only, it sleeps a random 100–500ms before running the procedure. On purpose. Locally your database is right next door and everything is instant, which hides request waterfalls — places where the UI fires request B only after request A resolves, so the two run in series instead of parallel. In production those serial round-trips add up to a visibly slow page; locally you'd never notice. The fake latency makes the waterfall painful during development so you fix it (by prefetching or parallelizing) before it ships. In production t._config.isDev is false, so there's zero added delay.

Gotcha

First-time reaction is always 'why is my dev API artificially slow?!' It's intentional. Don't rip it out — it's a design tool that makes production-only latency bugs visible locally. It's a no-op in production.

Note

next() is what actually runs the procedure (or the next middleware in the chain). Code before next() is pre-processing; code after is post-processing — same shape as Express/Koa middleware, but fully typed.

Why it's built this way

Waterfalls are invisible on localhost and brutal in production. Baking the latency into dev turns a class of hard-to-catch performance bugs into something you feel immediately, which is exactly when it's cheapest to fix.

publicProcedure vs protectedProcedure: auth as a type-level guarantee

This is the most elegant piece to understand, because it shows how middleware can change types, not just run code.

publicProcedure is just t.procedure.use(timingMiddleware) — the base procedure plus timing. Anyone can call it; ctx.session might be null.

protectedProcedure adds a second middleware. It checks ctx.session?.user; if there's no logged-in user it throws TRPCError({ code: 'UNAUTHORIZED' }), which stops the request cold with a 401. But look at what it passes to next(): a new ctx where session is spread with user: ctx.session.user. Because that runs only after the null-check, TypeScript now narrows ctx.session to non-null for the rest of the procedure. That's why every protected procedure body can write ctx.session.user.id with no ?. and no error — the middleware has proven, at the type level, that the user exists.

So the choice of base procedure is simultaneously your authorization policy and a compile-time promise about what ctx contains. In this codebase essentially every domain procedure is protectedProcedure (it's a single-teacher app; you must be logged in to touch any data). You reach for publicProcedure only for things that must work logged-out — and there are almost none here.

Why

Authorization lives in one middleware, not scattered if (!session) return 401 checks in every handler. Forget to use protectedProcedure and the endpoint is public — so the safe default in this app is: reach for protectedProcedure unless you have a specific reason not to.

Analogy

It's like a TypeScript type guard (if (x) { /* x is narrowed */ }) that persists for the entire procedure. The runtime throw and the compile-time narrowing are the same line of reasoning.

Why it's built this way

Encoding 'is authenticated' into the type of ctx means the compiler enforces it: you literally cannot write a protected procedure that treats the user as maybe-null, and you cannot forget the auth check because it's baked into the procedure you chose.

root.ts: composing the app router and exporting the type that powers everything

root.ts is short but load-bearing. It imports every feature router (student, lesson, report, user, piece, earnings, payment) and merges them into one appRouter via createTRPCRouter. The keys you give here are the namespaces you'll use on the client: because it's { student: studentRouter, ... }, the client calls api.student.getAll and api.payment.getForMonth. Adding a new feature router is literally one import + one line here.

Then the single most important line in the whole tRPC setup:

export type AppRouter = typeof appRouter;

This exports the type of the entire API — every router, every procedure, every input and output shape — as one big TypeScript type. It's typeof, so it carries zero runtime code; it evaporates at compile time. This is the contract. The client's createTRPCReact<AppRouter>() and the RSC createHydrationHelpers<AppRouter>() both consume this type, and from it TypeScript reconstructs the entire typed api object with correct inputs and outputs. When you change a procedure's return shape, AppRouter changes, and every call site that's now wrong lights up red. No codegen, no rebuild step — it's just TypeScript inference following the import.

Finally, createCaller = createCallerFactory(appRouter) produces a factory for calling procedures server-side given a context — this is what src/trpc/server.ts uses to give Server Components direct access.

Note

The object keys here ARE the client namespaces. { student: studentRouter } is why you call api.student.getAll and not api.studentRouter.getAll. Rename the key and every call site's namespace changes.

Why

export type AppRouter = typeof appRouter is the entire 'no codegen' story. The type is derived from the real router value, so it's impossible for the type to drift from the implementation — they're the same object.

Why it's built this way

Keeping composition (root.ts) separate from the tRPC plumbing (trpc.ts) means adding a feature never touches the framework wiring — you write a router file and register it in one place, and the type contract updates itself.

The client wiring for browser components (src/trpc/react.tsx)

Client Components talk to the server over HTTP, and this file builds the typed React-Query hooks that do it. Three things happen.

export const api = createTRPCReact<AppRouter>() creates the client-side api object, typed from AppRouter. This is what every "use client" component imports to get api.student.getAll.useQuery() etc. It also exports RouterInputs/RouterOutputs helper types so you can name a procedure's input or output type anywhere (e.g. RouterOutputs['student']['getAll']).

TRPCReactProvider is a context provider (mounted app-wide via src/components/providers/index.tsx). It sets up the tRPC client with a chain of links — links are like middleware for outgoing requests, composed in order: - loggerLink logs requests to the console in development (and logs downstream errors always). - httpBatchStreamLink is the transport. httpBatch means multiple procedure calls fired in the same tick get batched into one HTTP request to /api/trpc — so the calendar page requesting lesson.getInRange and student.getAll together costs one round-trip, not two. The Stream variant streams responses back as each resolves rather than waiting for the slowest. Note transformer: SuperJSON here — this is the client half that must match the server. It also stamps an x-trpc-source: nextjs-react header so the backend can tell where a call originated.

The QueryClient (React Query) is created via getQueryClient, which uses a singleton in the browser but a fresh client on the server — so server-render state never leaks between requests. Its config lives in query-client.ts: a 5-minute staleTime, superjson (de)serialization for hydration, and a QueryCache.onError that pops a toast on load failures (browser only).

Note

tRPC's React hooks ARE React Query hooks under the hood. Everything you know about React Query — staleTime, invalidate, placeholderData, enabled — works, because tRPC just wraps it with typed keys.

Why

Batching (httpBatch...) is why firing several useQuerys on one page doesn't mean several HTTP requests. They coalesce into a single POST to /api/trpc, which the batch handler fans out server-side. Fewer round-trips, faster pages.

Gotcha

A fresh QueryClient on the server, singleton in the browser (getQueryClient). If you accidentally shared one server-side, one user's data could bleed into another's render. The typeof window === 'undefined' branch prevents that.

Why it's built this way

Layering tRPC on React Query means the app gets caching, dedup, background refetch, and mutation lifecycles for free, while tRPC contributes the types and the batching transport — you don't hand-write fetchers or query keys.

The server-side caller for RSCs (src/trpc/server.ts) and the HTTP handler

Server Components run only on the server, so making them do an HTTP round-trip back to their own /api/trpc would be silly. src/trpc/server.ts instead gives them a direct caller. The file starts with import "server-only" — a guard that makes the build fail loudly if this module is ever imported into client code (it holds server secrets like the db).

It builds createContext (wrapping createTRPCContext with the RSC's headers from next/headers, tagged x-trpc-source: rsc, and cache()-wrapped so it runs once per request), then createCaller(createContext) produces a caller, and createHydrationHelpers<AppRouter> exports two things: api (the RSC-side caller) and HydrateClient (a component that dehydrates prefetched React Query state and streams it to the browser). So an RSC can await api.student.getAll() — a plain in-process async call, no network, no serialization — and get a fully typed result.

The HTTP handler at src/app/api/trpc/[trpc]/route.ts is the other door — the one browser requests come through. It's a normal Next.js App Router route handler exporting GET and POST. It hands the request to tRPC's fetchRequestHandler, pointed at appRouter, with a createContext built from req.headers, plus an onError that logs failures (and prints them in dev). The [trpc] dynamic segment is how one file serves every procedure: the path after /api/trpc/ tells the handler which procedure(s) to run.

So: two doors into the exact same appRouter. RSCs walk through server.ts (direct function call). Browsers walk through route.ts (HTTP). Both produce identical ctx, run identical procedures, and yield identically typed results.

Gotcha

There are TWO api imports and they are NOT interchangeable. import { api } from '@/trpc/server' is the async RSC caller (await api.x.y()). import { api } from '@/trpc/react' is the hooks object (api.x.y.useQuery()). Import the wrong one and nothing type-checks.

Why

import 'server-only' is a real safety net: the RSC caller closes over db and secrets. If someone imports it into a 'use client' file, the build breaks instead of shipping your database client to the browser.

Note

cache() (React's per-request cache) makes createContext run at most once per request even if several RSCs call procedures — so the session/db aren't re-resolved repeatedly during one render.

Why it's built this way

Direct in-process calls for RSCs skip HTTP, serialization, and an extra network hop entirely — a Server Component fetching data is as cheap as a function call, while Client Components still get a real HTTP API. One router, two optimal transports.

Call style #1 — Client Components: useQuery and useMutation in the wild

In a "use client" component you import api from @/trpc/react and call procedures as React Query hooks. This is the interactive path — anything that reacts to user state, refetches, or mutates.

Reads use .useQuery(input, options). In calendar/page.tsx, api.lesson.getInRange.useQuery({ start, end }, { placeholderData: keepPreviousData }) returns { data, isPending, isError, refetch }. The input { start, end } is type-checked against the procedure's Zod schema, and data (typed Lesson[]) needs no annotation — it's inferred from the procedure's return. keepPreviousData is a standard React Query option (tRPC passes options straight through) that keeps last month's grid on screen while the next month loads. On the same page api.student.getAll.useQuery() takes no argument, because that procedure declares no input — and TypeScript enforces that too.

Writes use .useMutation(callbacks), which returns a mutate/mutateAsync function plus lifecycle callbacks. The student-form.tsx create mutation shows the standard pattern in this repo: onSuccess toasts and closes the dialog after the server confirms (so a failed submit keeps the form open), onError toasts error.message, and onSettled calls utils.student.getAll.invalidate() to refetch the list. utils = api.useUtils() is tRPC's typed handle to the React Query cache — utils.student.getAll.invalidate() invalidates exactly that query's cache by its typed key, no magic string. router.refresh() additionally re-runs the RSC that server-rendered the page.

The student-form.tsx getByGuid.useQuery({ id: studentId! }, { enabled: !!studentId }) shows a conditional fetch: enabled: false tells React Query not to run until you're in edit mode (there's a studentId).

Tip

api.useUtils() is your typed remote control for the cache: utils.<router>.<proc>.invalidate() / .setData() / .prefetch(). It replaces React Query's stringly-typed query keys with autocompleted, refactor-safe paths.

Note

The repo's consistent mutation convention: close/reset the UI in onSuccess (only after the server confirms), surface failures in onError via error.message, and invalidate + router.refresh() in onSettled. Following it keeps optimistic-looking UIs honest.

Gotcha

useQuery runs on mount by default. Use { enabled: false } (as in the edit-mode fetch above) to gate a query on some condition, or you'll fire requests with placeholder/undefined inputs.

Why it's built this way

Because tRPC returns real React Query hooks, the team writes zero fetch/axios code and zero query-key constants. The typed utils object means cache invalidation can't silently target the wrong key — a common source of stale-UI bugs in hand-rolled setups.

Call style #2 — Server Components: await the procedure directly

The other path is the App Router's default: a Server Component (no "use client") is an async function that runs on the server, so it can just await a procedure through the RSC caller and pass the result down as props. No hook, no loading state, no HTTP.

students/page.tsx is the canonical example. It imports api from @/trpc/server, does const students = await api.student.getAll(), and renders <StudentsTable data={students} />. That call runs in-process, hits the same protectedProcedure, and returns fully typed data — students is typed exactly as the procedure returns, and passing it to the client StudentsTable (whose prop type must match) is checked at compile time. pieces/page.tsx, lessons/page.tsx (which calls two procedures, student.getAll and lesson.getAll({ from, to })), reports/page.tsx, and payments/page.tsx all follow the same shape.

Why do this instead of a client useQuery? Because it renders on the server with the data already present — no client-side loading spinner, faster first paint, and the data is baked into the HTML. The pattern here is: RSC fetches the initial data and hands it as props to a Client Component, which then owns interactivity (sorting, dialogs, mutations) and can re-fetch via useQuery/invalidate as the user acts. You get SSR's fast first render and SPA interactivity, split cleanly along the server/client line.

(For pages that want the client hook to also start already-warm, the toolkit provides HydrateClient + void api.x.prefetch() — the RSC prefetches into the query cache and streams it so the matching useQuery on the client hydrates without a second fetch. src/app/page.tsx uses HydrateClient. The direct-await-and-pass-props style above is the simpler, more common approach in this repo.)

Analogy

An RSC calling await api.student.getAll() is closer to a backend controller calling a service method than to a browser calling an API. There's no network between them — the 'API call' is a function call.

Why

Fetch-in-RSC-then-pass-as-props means the list is in the initial HTML (fast first paint, SEO-friendly), while the Client Component layered on top owns all the interactivity. It's the App Router's server/client split expressed through tRPC.

Gotcha

You can't call useQuery in a Server Component (hooks need the client), and you can't await api.x() the RSC caller inside a 'use client' component. Which api you import (/trpc/server vs /trpc/react) must match which kind of component you're in.

Why it's built this way

Splitting each page into an RSC that fetches + a Client Component that interacts is the App Router's core pattern, and tRPC makes both sides consume the same typed procedures — so there's one source of truth for data-fetching whether it happens on the server or in the browser.

How the types actually flow end to end

Tie it together by tracing a single type from database to JSX. This is the mental model that makes tRPC click.

  1. In student.ts, getAll returns ctx.db.student.findMany({ include: {...} }). Prisma types that return value precisely from your schema — it's a Student[] with the exact included relations and _count. You annotate nothing; the return type is inferred.
  2. getAll is a property of studentRouter, which is a property of appRouter. So the shape of getAll's output is embedded in typeof appRouter.
  3. root.ts exports type AppRouter = typeof appRouter — a compile-time-only description of every procedure's input and output.
  4. On the client, createTRPCReact<AppRouter>() (in react.tsx) and createHydrationHelpers<AppRouter>() (in server.ts) take that type and reconstruct the whole typed api object. api.student.getAll.useQuery()'s data is inferred to be exactly the Prisma return type from step 1. Inputs are inferred from each procedure's .input() Zod schema, so calling getByGuid without { id } — or with a wrong-typed id — is a compile error.

The key insight: nothing is generated and nothing is duplicated. The client doesn't have a copy of the types; it imports the type and lets TypeScript's inference walk it. That's why editing a procedure's return in student.ts instantly changes what .useQuery().data is on the client, and any now-broken consumer shows a red squiggle before you ever run the app. The runtime cost is zero (types compile away); the transport is superjson-over-httpBatchStreamLink for browsers and a direct function call for RSCs; and validation + auth are enforced by .input() schemas and protectedProcedure. REST gave you a string and a prayer; tRPC gives you a function signature the compiler checks on both ends.

Tip

Need the type of a procedure's output somewhere (a prop, a helper)? Use RouterOutputs['student']['getAll'] instead of redeclaring a Student interface by hand — it tracks the procedure automatically and never drifts.

Why

Prisma → procedure → AppRouter → client is one unbroken inference chain. A DB schema change ripples through Prisma's types, into the procedure's inferred return, into AppRouter, into every call site — surfacing as compile errors, which is exactly where you want to find them.

Gotcha

This chain is only as strong as the boundary. Casting inside a procedure (as any, as Foo) or annotating a useQuery result by hand breaks inference and reintroduces the exact drift tRPC exists to prevent. Let the types flow; don't override them.

Why it's built this way

Type-import-plus-inference (rather than codegen) means there is never a stale generated file, never a 'did you re-run the generator?' bug, and never a build step between changing the server and seeing the client update. The compiler is the codegen.

Key takeaways

  • tRPC lets you call server procedures as if they were local typed functions — no REST endpoints, no schema files, no codegen. You write routers of query/mutation procedures and export one AppRouter type that the client infers everything from.
  • .input(zodSchema) does double duty: it validates the request at runtime AND infers the input's TypeScript type at compile time, from a single Zod schema that's often shared with the front-end form.
  • src/server/api/trpc.ts is the plumbing: createTRPCContext injects { db, session, headers } into every procedure; initTRPC is configured with the superjson transformer (so Dates survive the wire) and an errorFormatter that logs real server crashes and surfaces Zod validation as typed zodError data.
  • publicProcedure vs protectedProcedure: the protected middleware throws UNAUTHORIZED for anonymous callers and re-passes ctx with a non-null user, so ctx.session.user.id is type-safe everywhere. Nearly every procedure in this app is protected.
  • The dev-only artificial 100–500ms delay in timingMiddleware is intentional — it exposes request waterfalls that a fast local DB would hide, and it's a complete no-op in production.
  • There are TWO ways to call the same procedure, and two different api imports. Client Components (@/trpc/react) use .useQuery()/.useMutation() hooks over batched HTTP to /api/trpc. Server Components (@/trpc/server) await api.x.y() as a direct in-process call — no network. Don't cross the imports.
  • The common page pattern: a Server Component fetches initial data (await api.student.getAll()) and passes it as props to a Client Component that owns interactivity and re-fetches via useQuery/utils.*.invalidate(). Fast first paint plus SPA interactivity, split along the server/client line.
  • root.ts's export type AppRouter = typeof appRouter is the whole contract. The client imports that TYPE (zero runtime cost) and TypeScript reconstructs the entire typed api — so a change to a procedure's return type instantly red-squiggles every broken call site with no build step.
  • The mutation convention across the repo: onSuccess closes/resets UI only after the server confirms, onError toasts error.message, onSettled calls utils.<proc>.invalidate() + router.refresh(). api.useUtils() is the typed, refactor-safe handle to the React Query cache.
  • Type safety is an unbroken chain: Prisma return type → procedure output → AppRouter → client useQuery().data. Casting (as any) or hand-annotating results anywhere along it breaks the inference tRPC exists to provide.
Chapter 07

Prisma & the data model

One schema file becomes a fully-typed database client. Here's every model in this app, how they relate, and the real query patterns.

If you've only ever done client-side data fetching in React, you talked to a backend someone else wrote — you fetch('/api/students') and got JSON. Prisma is the tool that backend uses to talk to the database. It is an ORM (Object-Relational Mapper): instead of writing raw SQL strings like SELECT * FROM "Student" WHERE ..., you call typed JavaScript methods like db.student.findMany({ where: { teacherId } }) and get back plain JS objects. The "typed" part is the magic — Prisma reads your schema and generates a client where every model, field, and relation is known to TypeScript, so autocomplete and compile-time errors catch mistakes before they ever hit the database.\n\nThe workflow is a straight line: you describe your data in one file, prisma/schema.prisma (models, fields, relationships), then run a generate step that produces @prisma/client — a TypeScript library shaped exactly like your database. Import it, and db.lesson.create(...) knows a Lesson has a rate (Int) and a status (the LessonStatus enum) and a student relation. Change the schema, regenerate, and the types update. Think of it as the difference between fetch returning any versus a fully-typed SDK someone hand-wrote for your exact API — except Prisma writes the SDK for you.\n\nThis project models a piano teacher's world: a Teacher has Students, each student has Lessons (scheduled, completed, or cancelled), Pieces they're learning, monthly Reports, and a payment system (PaymentMonth + PaymentTransaction). On top of that sit four tables — User, Account, Session, VerificationToken — that you did not design; NextAuth (the auth library) requires them and Prisma just hosts them. We'll walk every model.\n\nTwo project-specific things matter a lot and will bite you if you don't know them. First, the database is Neon Postgres, and this repo drives schema changes with prisma db push, not prisma migrate deploy — the migration history is out of sync, so running migrate against production would break things. Second, before you regenerate the client you must stop the dev server, because on Windows the running Node process locks the generated files and prisma generate fails. Both are covered in detail below.\n\nOne more foundational idea before we dive in: the whole app funnels every database call through a single shared Prisma client exported from src/server/db.ts. There's a small but important trick there involving a global variable that only exists to make Next.js's hot-reload in development behave. We'll unpack why.

What Prisma actually is: schema → generated client → typed queries

Prisma has three moving parts and understanding the flow between them removes almost all the mystery.

1. The schema (prisma/schema.prisma) is the single source of truth. It's written in Prisma's own small language (not SQL, not JS). At the top, two blocks configure everything: the generator block says what to produce (a JS client) and for which platforms, and the datasource block says what database and where — here PostgreSQL, with the connection string read from the DATABASE_URL environment variable. Notice it never hardcodes the URL; env("DATABASE_URL") pulls it at runtime.

2. Generation. When you run prisma generate (this repo also runs it automatically via the postinstall script), Prisma reads the schema and writes a TypeScript library into node_modules/@prisma/client. This library has a class PrismaClient with a property for every model — .user, .lesson, .paymentMonth — each carrying fully-typed methods. This is code generation: the types literally match your schema because they were built from it.

3. Querying. You import the generated PrismaClient, instantiate it once, and call methods. Because the types were generated from the schema, db.student.findMany({ where: { teacherId } }) autocompletes teacherId, rejects where: { teecherId } at compile time, and types the return value as an array of fully-shaped Student objects.

The binaryTargets line exists because Prisma ships a native query engine binary; "native" covers your local machine (Windows) and "rhel-openssl-3.0.x" covers the Linux runtime it deploys to. Without the second, the deployed app couldn't talk to the DB.

Analogy

Prisma's generated client is like a bespoke, fully-typed SDK for your exact database — the thing you wish every REST API shipped. You don't write it; the schema does.

Why

The URL comes from env("DATABASE_URL") rather than being hardcoded so the same schema points at your local DB, CI, and production just by swapping an environment variable — secrets never live in the repo.

Why it's built this way

A single declarative schema means the database shape, the TypeScript types, and the query API can never silently drift apart — they're all derived from one file. That's the core ORM value proposition and why T3 projects lean on it.

The database client singleton, and why a global in dev

Every query in this app goes through one exported object: db, from src/server/db.ts. Creating a PrismaClient opens a pool of database connections, so you want exactly one instance for the whole server process — not a new one per request, and definitely not per component.

The subtle part is the globalForPrisma dance. In production the file simply creates one client and exports it — clean. But in development, Next.js hot-reloads your code constantly: every time you save a file, modules get re-evaluated. If db.ts naively did export const db = new PrismaClient(), each hot-reload would build a fresh client and open another connection pool, and after a dozen saves you'd exhaust the database's connection limit and see too many connections errors.

The fix: stash the client on the globalThis object, which hot-reload does not wipe. On first load, globalForPrisma.prisma is undefined so it creates a client; on every subsequent reload, globalForPrisma.prisma ?? finds the existing one and reuses it. The final line only writes back to the global when not in production — in prod there's no hot-reload, so no global is needed and you avoid leaking the client onto global scope.

The log option is a nice touch: in development it logs every query, error, and warning to your terminal (great for seeing exactly what SQL Prisma generates), while production only logs errors to keep noise down.

Gotcha

Never do new PrismaClient() inside a request handler, tRPC procedure, or React component. Always import the shared db from src/server/db.ts. Spawning clients per call will exhaust your Neon connection pool.

Why

globalThis survives Next.js hot-module-replacement; a normal module-level const does not. The global is a dev-only cache to stop connection leaks, not application state — hence it's skipped in production.

Why it's built this way

This singleton pattern is boilerplate the T3 stack ships by default precisely because the connection-exhaustion bug is so common and so confusing for newcomers. It's worth recognizing on sight.

The NextAuth tables: User, Account, Session, VerificationToken

These four models are the odd ones out — you didn't design them for the piano domain; the NextAuth / Auth.js Prisma adapter requires them to exist with these exact field names. Prisma just hosts the tables; the auth library reads and writes them.

User is the account root: id (a cuid() — a collision-resistant unique string, friendlier than a raw UUID), optional name/email/image (optional because different login providers supply different data), an optional password (this app also supports credentials login, hence a stored bcrypt hash), a timezone (domain-specific, used everywhere for date math), and back-relations accounts, sessions, and teacher. That last one — teacher Teacher? — is the bridge from the auth world into the app's world: a User optionally is a Teacher.

Account represents one external login method linked to a user (e.g. a Google OAuth account). It stores OAuth tokens (access_token, refresh_token, etc.). The key line is @relation(fields: [userId], references: [id], onDelete: Cascade) — delete a User and their Accounts vanish automatically. The @@unique([provider, providerAccountId]) guarantees the same Google account can't be linked twice.

Session is a server-side login session (sessionToken is unique, expires when it lapses), also cascading on user delete. VerificationToken backs email magic-links / verification flows; it has no id at all — its identity is the composite @@unique([identifier, token]).

As a newcomer, the practical takeaway: treat these four as managed by the library. You rarely query them by hand — NextAuth does. But they live in the same schema and same database so relations (User → Teacher) can be enforced by the DB.

Note

@default(cuid()) generates the ID in the app layer, not the database — a sortable-ish, URL-safe unique string. @id marks the primary key, @unique a uniqueness constraint, @@unique([a,b]) a composite one across two columns.

Gotcha

Don't rename or drop fields on these four models to 'clean them up.' The Auth.js adapter expects these exact names (including the snake_case OAuth fields like access_token). Changing them breaks login.

Why it's built this way

Co-locating auth tables in the same Prisma schema lets the database enforce the User→Teacher relationship and lets one db client serve both auth and domain queries, instead of running a second database or ORM just for sessions.

The domain core: Teacher, Student, Piece

Now the app's real models. Teacher is the tenant of this whole app — almost every other row is scoped to a teacher. It has userId @unique (one User = at most one Teacher) with a relation back to User, its own timezone, and back-relations to everything it owns: lessons, students, pieces, payments, paymentTransactions. When you see a query start with db.teacher.findUnique({ where: { userId: ctx.session.user.id } }), that's the app translating 'the logged-in user' into 'their teacher record' before touching any data — the multi-tenant gate.

Student belongs to a teacher (teacherId + relation). The domain-critical fields are the two rates: lessonRate (in-person price) and onlineLessonRate (online price), both Int defaulting to 0. Storing money as an integer (whole currency units) sidesteps floating-point rounding bugs. A student also has back-relations to lessons, reports, payments, and paymentTransactions. The @@index([teacherId]) is deliberate: because the app constantly filters students by teacher, an index on that column makes those lookups fast instead of scanning the whole table.

Piece is a musical work a teacher tracks (title, optional description/level/difficulty). It's scoped teacherId and also indexed on it, and it's referenced by lessons. Nothing exotic here — it's the reference data lessons point at.

The recurring pattern across all three: a foreign key column (teacherId), a @relation describing how to join, and an @@index on the column you filter by most. That trio is the backbone of the schema.

Note

A list-typed field like lessons Lesson[] is a virtual back-relation — there's no column for it in the database. It exists so you can include: { lessons: true } and let Prisma do the join. The real foreign key lives on the other model (Lesson.teacherId).

Tip

@@index([teacherId]) isn't decoration — it's a performance decision. Because every list query filters where: { teacherId }, the index turns a full-table scan into a fast lookup. Add indexes to the columns you filter/sort by, not to everything.

Why it's built this way

Two separate rate columns (in-person vs online) plus integer money reflect a real billing rule: the two lesson types cost different amounts and must never suffer rounding errors. The student holds the current price; the lesson will snapshot it (next section).

Lesson: the status enum, isOnline, and the frozen rate snapshot

Lesson is the busiest model and the most instructive. Walk the fields: studentId + teacherId (both foreign keys), date (a @db.Timestamptz — timezone-aware; more below), duration in minutes, status (the enum), isOnline (was this an online lesson?), rate (the price charged for this specific lesson), actualMin (how long it actually ran, nullable — only known after it happens), and optional cancelReason, note, pieceId.

The enum. status is a LessonStatus, a Prisma enum with exactly three values: PENDING, COMPLETE, CANCELLED. An enum means the database itself refuses any other value — you can't typo "Completed" into the column. It defaults to PENDING (a freshly scheduled lesson). In queries you filter where: { status: "COMPLETE" } and TypeScript autocompletes the three options.

The rate snapshot — the single most important design idea in this schema. When a lesson is created, the app copies the student's current applicable rate onto Lesson.rate and freezes it there. Look at the create mutation: rate: effectiveLessonRate(student, input.isOnline). Why copy instead of reading student.lessonRate at report time? Because if the teacher raises a student's rate next year, last year's completed lessons must still show what was actually charged then. Reading the live student rate would silently re-price all of history. The snapshot makes the past immutable. (This is important enough that it's recorded in the project's memory notes.)

The relations differ in onDelete. Deleting a Student or Teacher cascades — their lessons are meaningless without them. But piece uses onDelete: SetNull: delete a Piece and the lesson survives with pieceId set to null, because the lesson still happened even if you stop tracking that piece.

Three indexes, each purposeful: @@index([teacherId, date]) and @@index([studentId, date]) are composite indexes — the app constantly queries 'this teacher's lessons between two dates' and 'this student's lessons between two dates,' and a composite index on (filter column, range column) serves exactly those. @@index([pieceId]) speeds the piece back-relation.

Why

onDelete: Cascade vs SetNull encodes what a row means: a lesson can't exist without its student (cascade), but it can exist without a tracked piece (set the FK null and keep the lesson).

Gotcha

Never compute earnings by reading the student's live rate. Sum Lesson.rate. Every earnings/payment query in this repo does exactly that — the frozen snapshot is load-bearing, and re-pricing past months is a bug.

Note

actualMin Int? is nullable because it's only known once a lesson is marked complete — until then there's no 'actual' minutes. The ? (nullable) vs no-? (required) distinction maps straight to NULL-ability in Postgres.

Note

The frozen-rate snapshot is powered by a tiny pure helper — effectiveLessonRate(student, isOnline) in src/lib/rate.ts — and it's written on every lesson write path (create, update, mark-attendance, recurring), not just create. Chapter 10 dissects this.

Why it's built this way

Snapshotting the rate per lesson is the schema-level embodiment of 'the past is immutable.' It trades a little denormalization (the price is stored redundantly on every lesson) for correctness: raising a rate today can never rewrite yesterday's invoices.

@db.Timestamptz: storing timezone-aware UTC

Every date column in this schema carries @db.Timestamptz. This tells Postgres to use its timestamptz (timestamp with time zone) type rather than a naive timestamp. The practical effect: values are stored as an absolute instant in UTC, and the database won't silently reinterpret them against whatever timezone the server happens to run in.

Why this matters intensely for a piano-lesson app: a teacher in New York schedules a 4pm lesson. That's a specific instant in time. If you stored a naive timestamp, the meaning of that stored value would depend on the timezone of whatever machine reads it — a lesson could 'move' between your laptop (local dev) and the Linux server (UTC). With timestamptz, the stored value is an unambiguous UTC instant; the app converts to the teacher's timezone (from User.timezone / Teacher.timezone) only at the edges for display and for computing month boundaries.

You can see the discipline throughout the routers: they never do where: { date: { gte: someLocalDate } } with a naive date. Instead they call helpers like getStartOfMonthUTC(month, year, timezone) and getEndOfMonthUTC(...) to compute the exact UTC instants that bound a calendar month in the teacher's timezone, then filter on those. The column type and the query helpers are two halves of one timezone strategy.

There's even a dedicated migration (20260306045621_convert_datetime_to_timestamptz) that converted the older naive columns to timestamptz — evidence the team hit the naive-timestamp trap and fixed it deliberately.

Gotcha

A 'month' isn't the same UTC range for everyone. October in New York starts 4-5 hours after October in UTC. That's why the code computes month boundaries through the timezone helpers instead of using new Date(year, month, 1) directly.

Analogy

timestamptz is like always storing timestamps in a universal currency (UTC) and only converting to 'local money' at the moment you show a price to a user — never storing the local amount and hoping everyone remembers the exchange rate.

Why it's built this way

Storing UTC and converting at the edges is the only reliable way to handle a multi-timezone scheduling app; the project even removed its naive timestamp columns via a migration once the ambiguity caused bugs.

MonthlyReport, PaymentMonth, PaymentTransaction: composite uniques and JSON

These three round out the domain and showcase two more Prisma features: JSON columns and composite unique constraints.

MonthlyReport is one report per student per calendar month: studentId, month (1-12), year, plus free-text summary/comments/nextMonthPlan/tuitionNote. Two things stand out. lessonMetadata Json @default("{}") is a JSON column — a flexible bag of per-lesson annotations (reason, color, isSpecial…) that would be awkward as rigid columns; the app validates its shape in the router with a Zod schema before writing. And @@unique([studentId, month, year]) is a composite unique key: the combination of the three columns must be unique, so a student can't accidentally get two reports for the same month. updatedAt DateTime @updatedAt auto-stamps on every write.

PaymentMonth is a per-student, per-month billing bucket: expectedAmount (what's owed), scoped by both studentId and teacherId, with its own @@unique([studentId, month, year]). Its two indexes — @@index([teacherId, month, year]) and @@index([studentId, year]) — match the two ways the app slices payments (all students in a month, one student across a year). It owns a list of transactions.

PaymentTransaction is a single payment received against a PaymentMonth: amount, optional method/note, a date. It carries paymentMonthId, studentId, and teacherId — note the studentId/teacherId are denormalized (repeated here even though you could reach them through paymentMonth) so the app can query and index transactions directly by teacher or student without a join. Everything cascades on delete of its parents.

The composite unique is what makes upsert possible (next section) — you need a unique key to say 'update the row matching this student+month+year, or create it.'

Note

A JSON column trades type-safety at the DB layer for flexibility. This repo compensates by validating lessonMetadata with a Zod schema in the report router before it ever reaches Prisma — the shape is enforced in app code instead.

Tip

@updatedAt (not @default) tells Prisma to overwrite the timestamp on every update automatically. You never set it by hand. @default(now()) only fires on insert; @updatedAt fires on every write.

Why

PaymentTransaction repeats studentId/teacherId even though they're reachable via paymentMonth. That denormalization exists purely so @@index([teacherId, date]) and @@index([studentId, date]) can serve direct 'all this teacher's transactions' queries without a join.

Why it's built this way

expectedAmount on PaymentMonth is a denormalized snapshot written when a transaction is added — but the payment router deliberately recomputes 'expected' live from completed lessons in some views (getStudentHistory) because the snapshot goes stale if lessons change afterward. The schema stores a convenient default; the code knows when to trust it.

Real query patterns: select/include, groupBy+_sum, aggregate, upsert, Promise.all

Now the payoff — how the routers actually query. Five patterns cover almost everything.

select vs include. Both shape what comes back. select is a whitelist — you get only the listed fields (select: { rate: true } returns objects with just rate, nothing else), which cuts payload and is faster. include pulls in related models on top of the full record (include: { student: true, piece: true } gives you the lesson plus its joined student and piece). You can nest a select inside an include to grab a relation but only some of its columns — the earnings router does include: { student: { select: { id, name, avatar, lessonRate } } }. Rule of thumb: select when you need a slice, include when you need related rows.

groupBy + _sum. To total each student's completed-lesson revenue in a month in the database (rather than fetching every row and summing in JS), the payment router uses groupBy: group by: ["studentId"], filter with where, and ask for _sum: { rate: true }. Postgres does the aggregation; you get one row per student with the summed rate.

aggregate. When you don't need grouping — just one total — aggregate with _sum: { rate: true } returns a single summed number. addTransaction uses it to compute one student's expected amount for a month.

upsert on a composite unique. This is where the @@unique([studentId, month, year]) pays off. upsert says 'find the row with this unique key; if it exists run update, else run create.' You reference the composite key by its generated name studentId_month_year. Both the report router and payment router use this to avoid a manual find-then-branch.

Promise.all batching. Independent queries shouldn't run one-after-another. Several procedures fire multiple db.* calls concurrently with Promise.all — e.g. getQuickInsights runs three queries at once, and getOverallSummary runs students/lessons/paymentMonths together — cutting total latency to the slowest single query instead of the sum.

Tip

Reach for groupBy/aggregate when you want the database to do the math. The earnings router sometimes fetches rows and sums in JS instead — fine for small result sets, but groupBy scales better as data grows.

Note

The composite-unique key name in upsert (studentId_month_year) is generated by Prisma from the fields in @@unique([studentId, month, year]), joined by underscores. That's how you address a multi-column unique in a where.

Why

Promise.all matters because these queries don't depend on each other. Awaiting them sequentially would make the endpoint as slow as the sum of all three; batching makes it as slow as the slowest one.

Note

Two mild gotchas for grepping: getOverallSummary lives in the payment router (routers/payment.ts), not earnings; and the multi-tenant gate (logged-in user → their Teacher row) is centralized as getTeacherOrThrow(db, userId) in the payment router, while other routers inline db.teacher.findUnique({ where: { userId } }).

Why it's built this way

These patterns keep the app both correct and fast: select/include trim payloads, DB-side aggregation avoids shipping thousands of rows to Node, upsert removes race-prone find-then-create logic, and Promise.all hides latency — all while staying fully typed end to end.

Migrations vs `prisma db push`: this project's real workflow

Prisma offers two ways to get your schema changes into the actual database, and this repo has a strong, non-obvious policy about which to use.

prisma migrate generates versioned SQL files in prisma/migrations/ (you can see this project's history there — ..._add_lesson_rate_and_online, ..._convert_datetime_to_timestamptz, etc.). migrate dev creates them locally; migrate deploy replays them against production in order. This is the 'proper' auditable way — when the history is intact.

prisma db push skips migration files entirely: it diffs your schema.prisma against the live database and mutates the DB to match, right now. Fast, no files, great for prototyping — but no version history.

This project's rule: use db push, and NEVER run migrate deploy. The reason (captured in the project's memory notes) is that the migration history is out of sync with the actual Neon Postgres database. The migrations/ folder still exists as a historical record, but the live DB has drifted from it — so replaying migrations with migrate deploy would try to apply changes that don't match reality and could corrupt or fail against production. The safe path is: edit schema.prisma, then npm run db:push (which runs prisma db push) to sync the DB.

The other practical gotcha: stop the dev server before generating. Regenerating the client writes files into node_modules/@prisma/client. On Windows, the running next dev process holds those files open, so prisma generate (and db push, which regenerates afterward) fails with a file-lock error. Workflow: stop next dev → run db push / generate → restart next dev.

Note the package.json scripts are slightly misleading: db:generate is mapped to prisma migrate dev and db:migrate to prisma migrate deploy — but per project policy you use db:push. Treat the migrate scripts as off-limits for this database.

Gotcha

NEVER run prisma migrate deploy (npm run db:migrate) against this project's database. The migration history is out of sync with the live Neon DB; replaying it can fail or corrupt data. Use npm run db:push.

Gotcha

Stop the next dev server before prisma generate or prisma db push. The running Node process locks the generated client files on Windows and the command will fail with an EPERM/file-in-use error. Stop → push/generate → restart.

Tip

prisma studio (npm run db:studio) opens a local GUI to browse and edit rows — handy for inspecting what a query actually wrote without leaving your machine.

Why it's built this way

db push over migrate is a pragmatic call for a solo-maintained app on Neon where the migration ledger already drifted: pushing the schema diff directly is reliable, whereas replaying a broken migration chain is not. The migrations folder is kept only as historical documentation of how the schema evolved.

Authentication

How login, sessions, and route protection work with NextAuth v5 — from the password check to the JWT to protectedProcedure.

If you come from plain React, you're used to auth living entirely in the browser: you POST to a /login endpoint, get a token back, stash it in localStorage, and attach it to every fetch. Next.js with the App Router flips this. Because pages are React Server Components that render on the server before any JavaScript reaches the browser, the server needs to know who you are the moment a request arrives — not after the page hydrates. NextAuth v5 (rebranded "Auth.js") is the library that answers "who is this request?" on the server, and this repo leans on that server-first model hard.

The mental shift is this: instead of a useAuth() hook that reads a token in the browser, you get one async function, auth(), that you await inside any Server Component, layout, API route, or tRPC context. It reads an HTTP-only cookie the browser can't touch, verifies it, and hands you a typed session object (or null). There's no token juggling in your components at all. The whole thing is wired up once in src/server/auth/ and then consumed everywhere by that single helper.

NextAuth v5 introduced a specific shape you'll see in this codebase: you call NextAuth(config) once and it returns a bundle — { auth, handlers, signIn, signOut }. auth() is the "who am I?" reader. handlers are the GET/POST functions you hand to the catch-all API route so the library can service its own endpoints (callbacks, CSRF, sign-out, etc.). signIn/signOut are server-side actions you call from Server Actions. This repo splits that into two files: config.ts (the pure options object, no secrets, edge-safe) and index.ts (the actual NextAuth() call). That split is a v5 convention worth understanding early.

This project authenticates users with email + password (a Credentials provider, passwords hashed with bcryptjs), stores the session as a signed JWT cookie (not a database session), and enriches that JWT with two custom fields — the user's id and their timezone — so every server request knows both without a database round-trip. A few things the scaffolding hints at but does not actually do: there is no Google OAuth wired up (there's a decorative "Continue with Google" button that does nothing and isn't even rendered), and there is no PrismaAdapter (the Account/Session tables exist in the schema as dormant T3 boilerplate but go unused because of the JWT strategy). I'll be explicit about that throughout so you don't go hunting for code that isn't there.

By the end you'll be able to trace a full round trip: a form submit → a Server Action → signIn("credentials") → the authorize() password check → a JWT minted via the jwt callback → a cookie set on the browser → and every subsequent request reading that cookie via auth() to gate pages (layout.tsx redirect) and API calls (protectedProcedure).

The v5 shape: one config, one NextAuth() call, four exports

NextAuth v5 centralizes everything into a single initialization. You build a config object, pass it to NextAuth(), and destructure the pieces you need. This repo deliberately splits that across two files.

config.ts exports a plain object, authConfig, typed with satisfies NextAuthConfig. Note it's satisfies, not : NextAuthConfig — this keeps the exact literal types (so TypeScript knows session.strategy is precisely "jwt", not the wider string) while still checking the object is a valid config. The config file imports nothing that would break in Next.js's Edge runtime except db — it's meant to be the portable, declarative half.

index.ts is where the machine actually starts: it calls NextAuth(authConfig) once and pulls out four things. auth is the session reader. handlers are the route handlers. signIn/signOut are the server-side mutation helpers. Everything else in the app imports from this one module, so there's a single source of truth for the auth instance.

Why split config from instance? In v5 the config.ts object is intentionally kept free of heavy/Node-only imports so it could be imported into Edge middleware (a common v5 pattern). This repo doesn't ship auth middleware, but it keeps the T3 shape so you can add one later without a refactor.

Why

config.ts (declarative options, edge-safe) is split from index.ts (the NextAuth() call that pulls in Node crypto) so the config can be reused in Edge middleware later. That's the standard v5 / T3 layout, not an arbitrary choice.

Gotcha

satisfies NextAuthConfig is not the same as : NextAuthConfig. The satisfies keyword validates the shape but keeps strategy: "jwt" as the literal "jwt" instead of widening it to string. If you change it to a colon annotation you can lose that precision.

Correction

Careful with a common v5 talking point: in this repo config.ts is not Edge-safe. Its authorize() imports the Prisma client (db) and bcryptjs directly, both Node-only. The config.ts / index.ts split here mirrors the T3 file layout, but it would need refactoring before you could run auth in Edge middleware.

Why it's built this way

The two-file split (pure config vs. instantiated NextAuth) is the canonical v5 pattern that makes the config portable to the Edge runtime. This repo keeps it even though it doesn't yet use middleware, so adding route-level auth middleware later is a drop-in.

The Credentials provider: verifying an email + password with bcryptjs

This app has exactly one working provider: CredentialsProvider. A Credentials provider means you own the verification logic — NextAuth doesn't redirect anywhere or talk to a third party; it just calls your authorize() function with whatever the user typed and trusts what you return.

The credentials object (email/password) is metadata describing the fields — NextAuth can auto-generate a form from it, but this app ignores that and uses its own React forms. The real work is in authorize(credentials). It: (1) bails to null if either field is missing, (2) looks the user up by email in Postgres via Prisma, (3) bails if no user or the user has no password (e.g. an OAuth-only account), (4) compares the plaintext against the stored bcrypt hash with compare(), and (5) on success returns a small user object.

Returning null from authorize() is the signal for "reject" — NextAuth turns that into a CredentialsSignin error. Returning an object is the signal for "accept," and whatever fields you return here become the user argument to the jwt callback in the next step. That's why authorize deliberately returns timezone and image alongside id/email/name — those fields are the only chance to seed the JWT.

The password check uses compare from bcryptjs (a pure-JS bcrypt, no native build step — handy on Windows and serverless). Registration hashes with hash(password, 10); login verifies with compare(plaintext, hash). The 10 is the salt-rounds cost factor.

login form → signIn("credentials", {email, password})
                      │
                      ▼
        authorize(credentials)
          ├─ missing fields? ─────► return null (reject)
          ├─ db.user.findUnique(email)
          ├─ no user / no password ► return null (reject)
          ├─ compare(plain, hash)
          │     └─ mismatch ──────► return null (reject)
          └─ return { id, email, name, image, timezone }  ✔ accept
Gotcha

The fields you return from authorize() are the ONLY data that flows into the jwt callback's user argument. If you want something on the session (like timezone), you must return it here — there's no second chance to read it from the DB in the JWT strategy without an extra query.

Note

user?.password being null is treated as a rejection. That's a real case: a user row could exist with password = null (the schema makes it optional). It's the guard that would matter if OAuth accounts existed — but here it mostly just means malformed/incomplete users can't log in.

Analogy

Think of authorize() as the hand-rolled POST /login handler you'd write in an Express SPA backend — look up user, compare hash, return the user. NextAuth just wraps a cookie/JWT layer around it for you.

Why it's built this way

bcryptjs (pure JavaScript) is chosen over the native bcrypt package so there's no C++ build toolchain needed — important for Windows dev machines and serverless deploy targets like Vercel/Neon.

Session strategy: JWT, not database — and why the adapter is absent

NextAuth supports two ways to remember a logged-in user between requests:

  1. Database sessions: a random opaque token lives in the cookie; the real session data lives in a Session table. Every request does a DB lookup to resolve the token. Requires a database adapter (like PrismaAdapter).
  2. JWT sessions: the session data itself is encrypted/signed into the cookie as a JSON Web Token. No DB lookup per request — the server just verifies the signature with AUTH_SECRET.

This repo uses strategy: "jwt". That single line has big consequences. Because the session lives in a self-contained signed cookie, auth() can resolve a user with zero database queries — it just cryptographically verifies the cookie. That's why there is no PrismaAdapter anywhere in the config: the adapter's whole job is to persist Session/Account/User/VerificationToken rows, and the JWT strategy doesn't need a Session table at all.

Here's the confusing part for a newcomer: the Prisma schema does define Account, Session, and VerificationToken models. Those are dormant T3 boilerplate. Nothing writes to Account or Session in this app because there's no adapter and no OAuth. The only table auth actually touches is User (read in authorize, written in registration). Don't let those empty tables mislead you.

With strategy: "jwt" and CredentialsProvider, the adapter is in fact unsupported — NextAuth requires Credentials to use JWT sessions precisely because there's no OAuth account to persist. So this pairing is not just a preference; it's the only combination that works for username/password without extra machinery.

DATABASE strategy (NOT used here):
  cookie = opaque id  ──►  SELECT * FROM Session  ──►  user
                            (DB hit every request)

JWT strategy (used here):
  cookie = signed JWT  ──►  verify(AUTH_SECRET)  ──►  user
                            (no DB hit)
Why

JWT sessions mean auth() verifies a signed cookie instead of querying the DB on every request — cheaper and stateless, which suits serverless/Edge. The tradeoff: you can't instantly revoke a session server-side (the cookie is valid until it expires), and any data baked into the JWT is stale until the token is re-minted.

Gotcha

The Account, Session, and VerificationToken tables in schema.prisma are leftover T3 scaffolding. They're never written to here (no adapter, no OAuth). Seeing them does NOT mean database sessions are in use — the strategy: "jwt" line is the source of truth.

Note

pages: { signIn: "/login" } tells NextAuth to send unauthenticated redirects to this app's custom /login page instead of NextAuth's built-in generic form.

Why it's built this way

Credentials-based auth essentially requires the JWT strategy (NextAuth doesn't support the database-session strategy with the Credentials provider). So JWT + no adapter isn't a style choice — it's the supported path for email/password. The unused OAuth tables are just the untrimmed T3 starter schema.

Callbacks + module augmentation: getting id and timezone onto the session

By default a NextAuth session only carries name, email, and image. This app needs two more things on every request: the database id (to scope queries to the logged-in teacher) and the user's timezone (this is a scheduling app — every date must render in the teacher's zone). Getting custom fields onto the session in the JWT strategy is a two-callback relay race, plus a TypeScript declaration so the compiler knows about the new fields.

The jwt callback runs whenever a token is created or updated. On the initial sign-in, the user argument is populated (it's exactly what authorize() returned). This is the one moment to copy fields into the token: token.id = user.id, token.image, token.timezone ?? "UTC". On subsequent requests user is undefined, so the if (user) guard means it only seeds once, then the token carries those values forward. The token is what gets signed into the cookie.

The session callback runs every time auth() is called. It receives the decoded token and returns the session object your app sees. Here it spreads the token's id/image/timezone onto session.user. So the flow is: authorize → user → jwt writes to token → token is the cookie → session callback copies token back onto session.user.

Module augmentation is the TypeScript half. Out of the box session.user has no .id or .timezone, so TS would error. The declare module "next-auth" block extends the Session, JWT, and User interfaces to add these fields. This is purely compile-time — it changes types, not runtime behavior — but it's what makes ctx.session.user.timezone type-check across the whole codebase.

authorize() returns  →  { id, email, name, image, timezone }
        │  (this is the `user` arg)
        ▼
  jwt callback (sign-in only):  token.id/​image/​timezone = user.*
        │
        ▼   token is signed → cookie
  session callback (every auth() call):
        session.user.id/​image/​timezone = token.*
        │
        ▼
  your code:  session.user.id, session.user.timezone  ✔ typed via module augmentation
Gotcha

The if (user) guard in the jwt callback means custom fields are captured ONLY at sign-in. If a user changes their timezone in the app, their existing JWT still holds the old value until they log out and back in (or the token is otherwise re-minted). This is the classic staleness tradeoff of JWT sessions.

Note

?? "UTC" appears in three places (authorize's return is passed through, the jwt callback, and the session callback). It's defensive: a user with no timezone always resolves to UTC rather than undefined, so date math downstream never gets a bad zone.

Why

Baking timezone into the JWT means every server render knows the user's zone with zero extra DB queries — critical for a scheduling app that formats dozens of dates per page. The cost is the staleness above.

Note

Two small precisions: the literal ?? "UTC" fallback appears twice (the jwt and session callbacks), and the jwt callback seeds three fields into the token — id, image, and timezone. Also, User.timezone is String @default("UTC") in the schema (non-nullable), so a real row never lacks one — the ?? "UTC" guards defend the TypeScript type, not actual null DB values.

Why it's built this way

In the JWT strategy there's no per-request DB read, so the only way to make id/timezone cheaply available everywhere is to embed them in the token itself. The two-callback relay (write in jwt, expose in session) plus module augmentation is the standard, type-safe way to do that.

auth(), the catch-all route, and tRPC context: one reader, three call sites

Everything that needs "who is this?" funnels through the single auth() function. Three details make it work in the App Router.

1. cache() wrapping. In index.ts, auth is cache(uncachedAuth) — React's cache(), not a data cache. Within a single server request, a page's layout, the page itself, and the tRPC context might all call auth(). Without memoization each call re-parses and re-verifies the cookie. cache() dedupes them so the cookie is verified once per request. This is a small but real optimization the T3 template bakes in.

2. The catch-all API route. NextAuth needs its own HTTP endpoints — for the sign-in POST, sign-out, CSRF tokens, session polling, and OAuth callbacks. The file src/app/api/auth/[...nextauth]/route.ts is a four-line adapter: it re-exports the GET and POST from handlers. The [...nextauth] folder name is a Next.js catch-all segment, so any path under /api/auth/* (e.g. /api/auth/callback/credentials, /api/auth/session) is routed into NextAuth's internal machinery. You never write these endpoints yourself; you just wire the handlers in.

3. tRPC context. Every tRPC request builds a context via createTRPCContext, which calls await auth() and puts the resulting session (or null) into ctx alongside db. That's how the entire API layer gets the current user — every procedure can read ctx.session. Because auth() is cache()-wrapped, this shares the same verification the page render already did.

                     ┌──────────────────────────┐
  RSC / layout ──────►                          │
  page.tsx     ──────►   auth()  (cache-deduped) │──► verify cookie ──► session | null
  tRPC context ──────►                          │
                     └──────────────────────────┘

  browser hits /api/auth/*  ──►  [...nextauth]/route.ts  ──►  handlers.GET/POST
Note

[...nextauth] is a Next.js catch-all route segment. The folder name literally is the brackets-and-dots. It means one file services every /api/auth/... URL NextAuth invents internally — you never enumerate them.

Why

cache(uncachedAuth) uses React's request-scoped memoization so the layout, the page, and the tRPC context all share ONE cookie verification per request instead of redoing it three times. It's request-scoped — it does not leak between users or requests.

Tip

getServerAuthSession is just () => auth() — a friendlier alias re-exported for use in Server Components and pages. Both names hit the same cached function; use whichever reads better.

Note

There's also a client-side session layer this section doesn't touch: <SessionProvider> (in providers/index.tsx) exposes useSession() to Client Components, user-store-provider.tsx bridges that into the Zustand store, and logout is signOut({ callbackUrl: "/login" }) imported from next-auth/react (the client helper) — not the server signOut. See Chapter 9.

Why it's built this way

Funneling every 'who am I?' check through one cache()-wrapped auth() gives a single, deduplicated source of truth that works identically in Server Components, pages, and the API layer — no token passed around, no per-layer re-verification.

Gating access: protectedProcedure (API) and the layout redirect (pages)

There are two doors into this app — the API (tRPC) and the pages (RSC routes) — and each has its own guard, both reading the same session.

API side: protectedProcedure. tRPC procedures are built by chaining middleware. protectedProcedure adds a middleware that checks ctx.session?.user; if it's missing it throws a TRPCError with code UNAUTHORIZED. If present, it calls next() with a narrowed context where session and session.user are typed as non-null. That narrowing is the payoff: inside any protected procedure you can write ctx.session.user.id with no null-check and no ! — TypeScript knows it's there because the middleware guaranteed it. Every router that reads the current teacher builds on this.

Page side: the layout redirect. The (root) route group is the authenticated part of the app. Its layout.tsx is an async Server Component that awaits getServerAuthSession() and, if there's no session, calls redirect("/login"). Because this runs on the server before any child page renders, an unauthenticated visitor never receives protected HTML at all — they're 307-redirected to login. This is fundamentally different from an SPA <PrivateRoute> that renders and then checks; here the gate is upstream of rendering.

The mirror image lives on the auth pages: login/page.tsx and register/page.tsx do the inverse — if a session already exists, redirect("/dashboard"), so logged-in users can't sit on the login screen.

PAGE REQUEST → (root)/layout.tsx → await session
     ├─ null  → redirect("/login")
     └─ ok    → render children

API REQUEST → protectedProcedure middleware → ctx.session?.user
     ├─ missing → throw UNAUTHORIZED
     └─ present → next({ session narrowed to non-null })
Analogy

protectedProcedure is the tRPC equivalent of an Express requireAuth middleware — but with a superpower: on success it hands downstream handlers a context where the compiler knows the user is present, so no defensive if (!user) littering your resolvers.

Why

Guarding in the server layout.tsx (not in the client) means protected HTML is never even generated for a logged-out visitor. Compare an SPA <PrivateRoute> that mounts the page, then redirects — leaking a flash of protected UI. The App Router redirect happens before render.

Gotcha

The layout redirect protects every route nested under (root)/. If you add a new authenticated page, just place it in that route group — it inherits the guard automatically. Put it elsewhere and it's unprotected. Don't rely on the guard from the wrong folder.

Why it's built this way

Two guards for two surfaces: the layout redirect protects server-rendered pages before render, and protectedProcedure protects the data layer with type-narrowing so resolvers stay clean. Defense in depth — a page you forget to gate is still safe if all its data flows through protected procedures.

Registration and login walkthrough: Server Actions, zod, hashing, auto sign-in

The forms are ordinary client components ("use client") using react-hook-form + zod for client-side validation, but they don't call NextAuth directly. They package the fields into FormData and call a Server Action — an async function marked "use server" that runs on the server but is callable straight from the client like a normal async function. This is the App Router's replacement for hand-writing a /api/register fetch.

Registration (registerAction) does, in order: re-validate the raw form data with registerSchema.parse() (server-side validation — never trust the client's), check the email isn't already taken, hash(password, 10) with bcryptjs, detect the browser's timezone (defaulting to "UTC" if detection throws), and db.user.create() the new row. Then — the nice touch — it immediately calls signIn("credentials", { redirect: false }) with the same email/password so the new user is logged in without a second trip to the login page. It returns a plain { success, message, user } object; the client shows a toast and router.push("/dashboard").

Login (loginAction) is simpler: validate with loginSchema, call signIn("credentials", { email, password, redirect: false }). redirect: false is important — it makes signIn throw on bad credentials (an AuthError with type: "CredentialsSignin") instead of HTTP-redirecting, so the action can catch it and return a friendly "Invalid email or password." message. On success it re-fetches the user for the client and returns it.

One accuracy note that corrects a common assumption: registration creates only a User — it does not create a Teacher row. The Teacher record (which owns students, lessons, payments) is created lazily the first time the user does something that needs it, e.g. student.ts does teacher ??= await ctx.db.teacher.create(...). So a freshly registered user has a User but no Teacher until they act.

REGISTER: form → registerAction(FormData)
  parse(registerSchema) → email taken? → hash(pw,10) → db.user.create
     → signIn("credentials", redirect:false)  [auto-login]
     → return {success, user} → toast + router.push("/dashboard")
            (Teacher row created LATER, lazily, on first domain write)

LOGIN: form → loginAction(FormData)
  parse(loginSchema) → signIn("credentials", redirect:false)
     ├─ AuthError(CredentialsSignin) → {success:false, "Invalid email or password."}
     └─ ok → refetch user → {success, user} → toast + push("/dashboard")
Gotcha

redirect: false on signIn is deliberate. With the default (redirect: true), NextAuth throws a special redirect that Next.js follows, and you can't return a JSON result to the client. Setting it false makes signIn either resolve or throw an AuthError you can catch and translate into a toast message.

Note

The task brief said registration creates a 'user + teacher.' It does NOT — it creates only a User. The Teacher is created lazily via teacher ??= ctx.db.teacher.create(...) the first time the user creates a student or similar. Worth knowing when debugging a brand-new account with no teacher row yet.

Why

zod validation runs on BOTH client (react-hook-form's zodResolver) and server (schema.parse() in the action). The client copy is UX (instant feedback); the server copy is security (the client can be bypassed). Never rely on the client validation alone.

Tip

Timezone is detected with Intl.DateTimeFormat().resolvedOptions().timeZone. But a Server Action runs on the SERVER, so this reads the server's timezone, not the browser's — a subtle point. It's wrapped in try/catch and falls back to UTC. Just be aware the captured zone may not be the user's actual one depending on where the action executes.

Why it's built this way

Server Actions let the forms call server logic (hashing, DB writes, sign-in) as if they were local async functions — no hand-written REST endpoint, no manual fetch/CSRF wiring. Auto-signing-in after register removes a redundant login step. Lazy Teacher creation keeps registration minimal and defers domain setup until it's actually needed.

What the scaffolding implies but does NOT do: Google OAuth and AUTH_SECRET

Two pieces of the setup look like more than they are, and knowing the truth will save you hours.

Google 'OAuth' is not implemented. There's a GoogleButton component that renders a 'Continue with Google' button — but it's a plain styled <Button> with no onClick, no signIn("google") call, and grepping the codebase shows it's not imported or rendered anywhere. There is no GoogleProvider in the config, no GOOGLE_CLIENT_ID/SECRET in env.js, and no OAuth callback handling. So the only real way to authenticate is email + password. If someone asks 'where's the Google login handled?' — it isn't. The button is decorative/aspirational leftover.

AUTH_SECRET is the one env var that matters for auth. In env.js it's required in production but optional in development. This is the key NextAuth uses to sign and verify the JWT cookie — the entire security of the JWT strategy rests on it. In dev, NextAuth can auto-generate one, which is why it's optional locally; in production you must set a stable secret or every deploy would invalidate all existing sessions (and an empty secret would be insecure). Notice there are no AUTH_GOOGLE_* or NextAuth URL vars — further confirmation that OAuth was never wired.

The practical takeaway: this is a single-provider, password-only, JWT-session auth system. The Account table, the Google button, and the adapter-shaped schema are all dormant T3/starter artifacts. When extending auth, you'd add a real GoogleProvider to config.ts, wire the button's onClick to signIn("google"), add the Google env vars, and — because OAuth needs to persist accounts — likely add the PrismaAdapter (which would also pull the Account/Session tables into active use).

The project shipped the T3 starter's OAuth-shaped schema and a placeholder social button but only implemented the credentials path it actually needed. Leaving the scaffolding in place makes future OAuth cheaper to add, at the cost of some misleading dead code — hence this explicit callout.

Key takeaways

  • Auth is server-first: one auth() function (wrapped in React cache() so it verifies the cookie once per request) answers 'who is this?' inside Server Components, the (root) layout, and the tRPC context — there is no client-side token handling.
  • The session strategy is "jwt", meaning the session lives in a signed cookie and auth() needs zero DB queries. Consequently there is NO PrismaAdapter; the Account/Session/VerificationToken tables in the schema are dormant T3 boilerplate that nothing writes to.
  • The only working provider is CredentialsProvider. Its authorize() is the whole trust boundary: look up the user by email, bcryptjs.compare() the password, and return a user object (with id + timezone) or null. The returned fields are the only data that reaches the JWT.
  • Custom fields (id, timezone, image) get onto the session via a two-callback relay: the jwt callback seeds the token at sign-in only (if (user)), and the session callback copies them onto session.user on every read. declare module "next-auth" augments the types so this is type-safe. Downside: JWT data is stale until re-login.
  • Access is gated in two places reading the same session: pages via await getServerAuthSession() + redirect("/login") in (root)/layout.tsx (runs before render, no protected HTML leaks), and the API via protectedProcedure, which throws UNAUTHORIZED and type-narrows ctx.session.user to non-null for resolvers.
  • Forms call Server Actions, not NextAuth directly: registerAction re-validates with zod, hash(pw,10), db.user.create, then auto signIn("credentials", { redirect: false }). loginAction uses redirect: false so bad credentials throw a catchable AuthError (CredentialsSignin) instead of redirecting.
  • Correcting the brief: registration creates ONLY a User. The Teacher row (owner of students/lessons/payments) is created lazily on first domain write, e.g. teacher ??= ctx.db.teacher.create(...) in the student router.
  • Google OAuth is NOT implemented: GoogleButton has no onClick/signIn and is never rendered, there's no GoogleProvider, and env.js has no OAuth vars — only AUTH_SECRET (required in prod, optional in dev) which signs the JWT cookie.
  • The catch-all route src/app/api/auth/[...nextauth]/route.ts is a 4-line re-export of handlers (GET/POST); Next.js routes every /api/auth/* URL NextAuth invents into it.
Chapter 09

Client data, state & forms

Server data as a cache (TanStack Query), a sliver of client state (Zustand), forms with validation (RHF + Zod), and where errors go.

In a plain React SPA you usually reach for useState + useEffect + fetch to load data, and maybe Redux/Context for shared state. This app splits the job into two very different kinds of state, and it's worth internalizing the distinction before reading any code:

Server state is data that lives in the database and is merely cached in the browser — lessons, students, your profile. You don't own it; the server does. It can go stale, needs refetching, and must be invalidated after you change it. This app manages all server state with TanStack React Query, but you almost never call React Query directly — you go through tRPC, which gives you api.user.getProfile.useQuery() style hooks that are fully typed end-to-end (the return type is inferred from the server function). Think of tRPC as a typed RPC layer sitting on top of React Query's cache.

Client state is data that only the browser cares about — the currently-logged-in user object for quick UI reads, the selected currency, a modal's open/closed flag. That's handled by Zustand (a tiny global store) for the user, plain useState for local UI, and React Hook Form for the transient state inside a form while you're typing.

The mental model you must hold for React Query is the query cache: every useQuery is identified by a query key (for tRPC, the key is derived from the procedure path + input, e.g. user.getProfile with no input, or lesson.getAll with a filter object). The cache stores the last-known data per key, marks it fresh or stale based on staleTime, and refetches stale data on certain triggers. After a mutation (a write), you don't manually update every screen — you invalidate the relevant keys and React Query refetches them. That "read is a subscription to a cache key, write then invalidate the key" loop is the single most important idea in this whole topic.

On top of reads and writes sits an error-handling stack: a React ErrorBoundary for render crashes, a global QueryCache.onError that toasts failed loads, per-mutation onError callbacks that toast failed writes, and a set of server-side wrappers (api-error-handler, server-action-error-handler, validation-middleware, plus the tRPC errorFormatter) that shape errors into safe, typed messages before they ever reach the client. sonner (toast.*) is the single channel through which almost every error becomes visible to the user.

This document walks the whole path: how the QueryClient is configured, how reads/writes/invalidation work through tRPC, the optimistic-update pattern, the Zustand user store and its SSR-safety wrapper, the React Hook Form + Zod wiring, and finally how every kind of error surfaces.

The mental model: server state as a cache, keyed by query key

Before any code, lock in the model. React Query keeps an in-memory cache of query results. Each entry is addressed by a query key. In this app you never write raw keys — tRPC derives them for you from the procedure path and its input. api.user.getProfile.useQuery() reads the cache entry for ['user','getProfile']; api.lesson.getAll.useQuery(filters) reads the entry for ['lesson','getAll', filters]. Change the filters object and it's a different cache entry with its own data and loading state.

Each entry has a freshness clock governed by staleTime. While an entry is fresh, mounting another component that reads the same key returns the cached value instantly with zero network calls. Once it passes staleTime it becomes stale, and React Query may refetch it on triggers like remount or window refocus. This is why a plain-React developer is often surprised that navigating away and back doesn't re-hit the server — the cache answered.

Writes are the other half. A useMutation (a tRPC mutation) does NOT automatically know which reads it affected. The contract is: after a successful write, you invalidate the affected keys via the tRPC 'utils' object (utils.user.getProfile.invalidate()), which marks them stale and triggers a refetch, so the UI re-syncs with the database. Master that read→cache→write→invalidate loop and the rest is detail.

  Component renders
        |  api.x.y.useQuery(input)
        v
  [ Query Cache ]  key = ['x','y', input]
    fresh? --yes--> return cached data instantly
    stale? --------> return cached data + refetch in background
        ^
        |  utils.x.y.invalidate()  (after a mutation)
  [ useMutation writes to server ] --success--> invalidate keys
Analogy

Think of useQuery as useState whose source of truth is a shared, network-backed cache instead of local memory. Ten components reading the same key all see one value and re-render together when it changes — no prop drilling, no context you wrote.

Gotcha

The query key includes the input object by value. If you pass a freshly-built object each render with different contents, you fragment the cache. Keep inputs stable (memoize derived filter objects) so you actually hit the same entry.

QueryClient configuration: staleTime, a global error toast, and SSR-safe (de)hydration

createQueryClient() builds the single QueryClient the whole app shares. Three decisions here shape every query in the app.

(1) staleTime: 5 * 60 * 1000 — 5 minutes. The default React Query staleTime is 0, meaning every query is stale immediately and refetches aggressively. With server-rendered data that would cause an instant refetch on the client the moment the page hydrates (wasteful). Raising it to 5 minutes says 'trust freshly-loaded data for 5 minutes,' which suppresses that redundant refetch and makes navigation feel instant.

(2) A QueryCache with a global onError — instead of every useQuery call site writing its own error handling for loads, one place catches all query failures and shows a toast.error. The guard if (typeof window === 'undefined') return; matters: during SSR a per-request QueryClient also runs, and toast is browser-only, so it short-circuits on the server. The comment explains the deliberate choice NOT to log here — expected 401/404/validation errors would just be noise, and genuine server errors are already logged by the tRPC errorFormatter.

(3) dehydrate/hydrate with SuperJSON — for Server Components that prefetch data, React Query serializes the cache on the server ('dehydrate') and rebuilds it on the client ('hydrate') so the browser doesn't refetch what the server already loaded. Plain JSON can't carry Date, Decimal, etc.; SuperJSON can, which is why it's wired as the serializer here and as the tRPC transformer. The shouldDehydrateQuery override also ships still-pending queries so streaming SSR works.

Why

One QueryCache.onError means read failures surface as a toast automatically, everywhere, without each screen remembering to handle them. Write failures are handled separately in each mutation's onError (covered later) because writes usually want bespoke messages and rollback.

Note

SuperJSON appears in three places — the QueryClient (de)hydrate, the tRPC client link transformer (src/trpc/react.tsx), and the tRPC server init. All three must agree so a Date sent by the server arrives as a real Date in the browser.

One QueryClient per browser, a fresh one per SSR request

src/trpc/react.tsx wires the client-side tRPC + React Query provider tree. The subtle-but-important part is getQueryClient(). On the server it returns a brand-new QueryClient every call. In the browser it lazily creates one module-level singleton and reuses it forever.

Why the asymmetry? A single Next.js server process handles many users' requests concurrently. If the server reused one QueryClient across requests, User A's cached data could leak into User B's response — a serious data-isolation bug. So the server makes a throwaway client per request. The browser, by contrast, serves exactly one user for the life of the tab, so a persistent singleton is what you want — it's the cache that makes navigation instant.

The TRPCReactProvider also builds the tRPC client once via useState(() => ...) (lazy initializer, so it's created a single time, not on every render). Its httpBatchStreamLink batches multiple tRPC calls made in the same tick into one HTTP request and streams responses back; loggerLink logs calls in dev or on downstream errors. Finally it nests QueryClientProvider (React Query's context) inside which api.Provider (tRPC's context) lives — both must wrap the app for the typed hooks to work. This provider is mounted inside the app-wide Providers component.

Why

The ??= singleton in the browser is what lets you leave a page and come back without a network round-trip: the same cache is still there. Recreating the client on every mount would throw the cache away and defeat the whole point.

Gotcha

Server Components use a DIFFERENT entrypoint, src/trpc/server.ts (createHydrationHelpers, HydrateClient), not this client provider. Don't import api from @/trpc/react in a Server Component or from @/trpc/server in a Client Component — they are two separate tRPC surfaces.

Writes through tRPC: useMutation, useUtils, and invalidate-after-write

A tRPC mutation is api.<router>.<procedure>.useMutation(options). It returns { mutate, mutateAsync, isPending, ... } plus lifecycle callbacks onSuccess, onError, onSettled. mutate(input) fires and forgets (you react in callbacks); mutateAsync(input) returns a promise you can await.

The piece that ties writes back to reads is api.useUtils() — tRPC's typed wrapper around the QueryClient. utils.user.getProfile.invalidate() marks that query stale and refetches it. So the standard write flow is: call the mutation → on success, invalidate whatever reads it changed → toast success. On error, toast the message. The profile-form is the canonical example: after updateProfile succeeds it invalidates user.getProfile so the page re-reads fresh data, and toasts. Note error.message here is already a safe string — the server's error handlers (later section) guarantee it isn't a raw stack trace.

The students-table shows the same shape with router.refresh() added in onSettled. onSettled runs whether the mutation succeeded or failed — a good place for cleanup/refetch that should happen either way. router.refresh() is a Next.js App Router call that re-runs the Server Components for the current route, refreshing any server-rendered data that React Query doesn't own.

Tip

The { id: "student-delete" } passed to toast is a stable toast id. Reusing one id means a later toast REPLACES the earlier one instead of stacking — handy for optimistic flows where you show 'Deleting…' then swap it for success/failure.

Note

invalidate() can be scoped narrowly (utils.user.getProfile.invalidate()) or broadly (utils.lesson.invalidate() invalidates every query under the lesson router). Broader is simpler and safer against forgetting a key; narrower is cheaper. The codebase uses both deliberately.

Optimistic updates: cancel → snapshot → patch → rollback-on-error → invalidate-on-settled

For actions that should feel instant (deleting a lesson, toggling attendance), the app uses React Query's optimistic-update recipe directly on the tRPC utils. The pattern in lessons-page is the textbook five-step dance, and it's worth reading slowly because every step prevents a specific bug:

  1. onMutate runs before the request leaves. First await utils.lesson.getAll.cancel({}) — cancel any in-flight refetch so a slow older response can't land after our optimistic edit and clobber it.
  2. Snapshot the current cached data with utils.lesson.getAll.getData(filters) and stash it in the returned context. This is the undo buffer.
  3. Patch the cache immediately with setData(filters, old => old.filter(...)) — the row disappears from the UI instantly, before the server has even answered. It also fires the success toast and closes the dialog optimistically.
  4. onError(err, input, context) — if the server rejects, restore the snapshot with setData(filters, context.previousData). The row reappears; the user sees an error toast. No corruption.
  5. onSettled — regardless of outcome, reconcile with the server by invalidating. The guard queryClient.isMutating({ mutationKey }) ensures that when several deletes are in flight, only the last one to settle triggers the refetch (avoids a refetch storm).

Contrast this with the simpler profile-form flow (previous section) which just waits for the server then invalidates. Use optimistic updates when perceived latency matters and the operation rarely fails; use the simple flow otherwise.

Gotcha

Skipping the cancel() step is the classic optimistic-update bug: a background refetch that started before your edit can resolve after it and silently restore the deleted row. Always cancel first.

Why

The snapshot must be taken and returned from onMutate (not read again in onError) because by the time onError runs the cache has already been mutated — the only trustworthy copy of the pre-edit state is the one you saved.

Note

The same recipe appears in calendar/_components/attendance-dialog.tsx, which patches THREE caches at once (lesson.getAll, lesson.getInRange, earnings.getTodayLessons) because one attendance change affects three different screens.

Zustand: the user store and its session-syncing provider

Not everything is server state. The logged-in user's identity is read all over the UI (avatar, greetings, quick access to the user id), so it lives in a tiny Zustand store. create() returns a hook, useUserStore, that any component calls with a selector: useUserStore(s => s.setUser). No provider is needed for the store itself — Zustand keeps the state in a module-level closure. The persist middleware mirrors it to localStorage under the key user-storage, so a refresh restores the user without waiting on the network.

Be precise about what UserStoreProvider is and isn't. It is NOT the 'per-request store instance' pattern you may have read about (that pattern uses useRef/context to give SSR each request its own store). Here the store is a single global singleton. The provider is a small client component that does one job: read the NextAuth session (useSession()) and push it into the Zustand store whenever the user becomes authenticated. It's a bridge from the auth layer to the client store.

The isHydrated guard is the SSR-safety mechanism that matters here. Because persist reads localStorage (browser-only), the store's value on the server-rendered HTML and on the first client render can differ, which React flags as a hydration mismatch. Returning null until the first useEffect runs (i.e. until we're safely on the client) sidesteps that mismatch. The forms (login-form, register-form) also write to this store directly via setUser(result.user) right after a successful auth action, so the UI updates instantly without waiting for the session cookie round-trip.

        |                                                     useUserStore(selector)
  login/register action also calls setUser(result.user) directly          v

Correction to a common assumption: this app does NOT use the per-request-store-via-context pattern for Zustand. The store is a global singleton. React Hook Form + Zod + shadcn: how the form primitives wire together

Every form uses React Hook Form (RHF) for local form state and Zod for validation, glued by @hookform/resolvers/zod. useForm({ resolver: zodResolver(schema), defaultValues }) returns a form object. The key pieces: form.control (the connective tissue passed to fields), form.handleSubmit(onSubmit) (validates, then calls your handler only with valid, typed data), form.formState (isValid, errors, etc.), and form.reset() (repopulate values — used to sync in server-loaded data via useEffect). RHF keeps inputs uncontrolled under the hood for performance, so typing doesn't re-render the whole form.

The app never touches raw RHF register in these forms; instead it uses the shadcn form.tsx wrapper set, which is a thin, accessible layer over RHF's Controller. Read form.tsx to see the trick: Form is literally RHF's FormProvider. FormField wraps Controller and stashes the field name in a React context. useFormField() then reads that context plus useFormState to pull this field's error and generate matching id/aria-describedby/aria-invalid attributes. That's what makes FormMessage able to render the field's Zod error with zero wiring — it just calls useFormField() and prints error.message. FormLabel flips to the destructive color when the field has an error, and FormControl (a Radix Slot) forwards the accessibility ids onto whatever input you nest.

So a field is always the same four-part shape: FormField (binds control + name, gives you field) → FormItem (layout + id context) → FormLabel/FormControl(your <Input {...field}/>)/FormMessage. Spreading {...field} onto the input wires value, onChange, onBlur, name, and ref in one shot.

Why

The context indirection in form.tsx exists so FormMessage/FormLabel don't need the field name or error passed as props — they read it from context. That's what keeps every field's JSX short and identical, and keeps label/input/error correctly associated for screen readers.

Tip

zodResolver(schema) means the ONE Zod schema is your source of truth for both TypeScript types (z.infer<typeof schema>) and runtime validation. password-form even uses .refine() to cross-check that confirmPassword matches, with path: ['confirmPassword'] so the error attaches to the right field.

Gotcha

For non-native inputs (Checkbox, Select), you can't just spread {...field} — you manually map checked={field.value} onCheckedChange={field.onChange} (see login-form's remember checkbox) or value={field.value} onValueChange={...} (teacher-settings Select). Only plain <Input/> accepts a raw {...field} spread.

Two submit patterns: Server Actions (auth) vs tRPC mutations (profile)

The forms split into two submission styles, and knowing which is which prevents confusion.

Server-Action forms — login-form and register-form. On submit they build a FormData, call a Next.js Server Action (loginAction/registerAction from @/server/actions/*) directly as an async function, and branch on the returned { success, message, user } result. They manage their own isLoading with useState (no React Query here), toast the result, push to /dashboard, and importantly call setUser(result.user) to seed the Zustand store plus router.refresh() to re-pull server data now that a session cookie exists. Auth uses Server Actions rather than tRPC because it needs to set cookies/redirect at the framework level.

tRPC-mutation forms — profile-form, password-form, teacher-settings-form. These use api.*.useMutation() and get isPending, typed errors, and cache invalidation for free (covered earlier). No manual loading useState; the button reads updateX.isPending. teacher-settings-form is interesting: it fires the timezone mutation on change (from the Select's onValueChange) rather than on a submit button, and it also mixes in a non-server concern — currency is stored via a separate useCurrency() client hook, not the database — so that half of the form doesn't touch tRPC at all.

Both patterns funnel user feedback through the same toast channel, so from the user's perspective success/failure looks identical regardless of the underlying transport.

Note

mode: 'onChange' (password-form) validates on every keystroke so formState.isValid is live, which is why its submit button can be disabled until the whole form passes. The other forms validate on submit (RHF's default), which is cheaper.

Why

Auth is a Server Action, not a tRPC procedure, because sign-in must set the session cookie and coordinate a redirect at the Next.js framework layer — things tRPC's data-fetching model doesn't own. Ordinary CRUD (profile, password) has no such need, so it uses tRPC for the free typing + caching + invalidation.

Note

The auth Server Actions (loginAction/registerAction) don't use the generic wrapper — each has its own inline try/catch and returns { success, message, user }. That's why login-form checks result.success && result.user.

The error stack: from server wrappers to a toast the user actually reads

Errors are caught at several layers, each with a job, and almost all of them end at a sonner toast.

Server-side shaping (so the client only ever sees safe strings): - The tRPC errorFormatter (src/server/api/trpc.ts) runs on every failed procedure. It logs only genuine INTERNAL_SERVER_ERRORs (not expected 401/404/validation — those would be log noise) and attaches a flattened zodError to the response so the client can read field-level validation failures type-safely. - withErrorHandling (api-error-handler.ts) wraps REST route handlers: it logs, then maps custom error classes (ValidationError→400, AuthenticationError→401, AuthorizationError→403, NotFoundError→404) to proper status codes, and — crucially — in production replaces any unknown 500 message with a generic string so internals never leak. In development it passes the real message through for debugging. - withServerActionErrorHandling/tryCatch (server-action-error-handler.ts) do the same for Server Actions, returning a discriminated { success: true, data } | { success: false, error } object (again dev-vs-prod message masking). - validation-middleware.ts provides validateBody/validateQuery/withValidation for REST routes: safeParse against a Zod schema, and on failure return a readable path: message string with a 400.

Client-side surfacing: - ErrorBoundary (a class component, because only class components can catch render errors) wraps the whole provider tree. If a component throws during render, it logs via logError, shows a friendly fallback (with a stack trace only in development), and offers 'Try Again'/'Go Home'. This catches bugs, not data errors. - The QueryCache.onError toasts failed reads (section 2). - Each mutation onError toasts failed writes with a bespoke message. - useErrorHandler/useAsyncError (use-error-handler.ts) are a reusable hook for imperative try/catch spots: they log and toast, masking the message in production just like the server wrappers.

Underpinning all logging is logError (src/lib/error-handler.ts): it keeps a bounded in-memory ring buffer, console-logs in dev, and forwards to an external tracking URL if NEXT_PUBLIC_ERROR_TRACKING_URL is configured (the seam where Sentry would plug in).

  validation-middleware ───────────┘     QueryCache.onError ─> toast.error (reads)

                                         ErrorBoundary ──────> fallback UI (render crashes)

Why

The dev-vs-prod message masking (NODE_ENV === 'development' ? message : genericString) appears in api-error-handler, server-action-error-handler, and use-error-handler. It's a consistent security posture: engineers see real messages locally; users in production never see stack traces or internal detail that could aid an attacker.

Gotcha

An ErrorBoundary only catches errors thrown during React rendering — not errors inside event handlers, async callbacks, or useQuery/useMutation requests. Those are handled by QueryCache.onError, mutation onError, or useErrorHandler. Don't expect the boundary to catch a failed fetch.

Note

Key takeaways

  • Split state by ownership: server state (DB-backed, cached) goes through TanStack Query via tRPC; client state (current user, currency, UI flags) goes to Zustand or useState. Don't manage server data with useState+useEffect here.
  • The core loop is read via useQuery (keyed by procedure+input) → write via useMutation → invalidate the affected keys with api.useUtils() so reads refetch. React Query does not auto-sync writes to reads; you invalidate.
  • createQueryClient sets staleTime to 5 min (suppress redundant SSR refetch), a global QueryCache.onError that toasts failed loads, and SuperJSON (de)hydration so Dates/Decimals survive SSR. SuperJSON must match on client link and server init too.
  • One QueryClient singleton per browser tab (the persistent cache); a fresh one per SSR request (prevents cross-user data leakage). That asymmetry in getQueryClient is deliberate.
  • Optimistic updates follow a strict five steps: cancel in-flight refetches → snapshot cache → setData optimistically → rollback from the snapshot in onError → invalidate in onSettled. Skipping cancel() is the classic bug.
  • The Zustand user store is a GLOBAL singleton (not a per-request store). UserStoreProvider merely bridges the NextAuth session into it and gates render on isHydrated to avoid an SSR/localStorage hydration mismatch.
  • Forms = React Hook Form + zodResolver, wired through shadcn form.tsx. FormField/useFormField pass field name via context so FormMessage auto-renders the Zod error; the ONE Zod schema drives both TS types and runtime validation. Spread {...field} only onto plain inputs.
  • Two submit styles: auth uses Next.js Server Actions (need cookies/redirect), CRUD uses tRPC mutations (free typing + isPending + invalidation). Both report through sonner toasts.
  • The error stack shapes errors on the server (errorFormatter, api-error-handler, server-action-error-handler, validation-middleware) with dev-vs-prod message masking, and surfaces them on the client via ErrorBoundary (render crashes only), QueryCache.onError (reads), mutation.onError (writes), and useErrorHandler (imperative). Everything funnels to logError, the Sentry seam.
Chapter 10

The domain logic

The business rules that make the money correct: frozen per-lesson rates, live-recomputed payments, and timezone-aware months. This is the payoff chapter.

This is the part of PianoDiary that has to be correct, not just pretty. A piano teacher records lessons, marks who showed up, expects money each month, and records payments as they trickle in. The tricky part is that all of these numbers change over time — rates go up, lessons get cancelled, students pay late — and the app must never accidentally rewrite history. A report you generated for February must still say the same thing in July.

There are three big ideas woven through every file in this topic. First: the frozen per-lesson rate. Each Lesson row stores its own rate (an integer, in Vietnamese đồng) captured at the moment the lesson is created. The student carries two current rates — lessonRate for in-person and onlineLessonRate for online — but those are just the source used to stamp a lesson. Once stamped, a lesson keeps its rate forever unless you deliberately re-touch it. This is why a raise this month never re-prices last month.

Second: expected money is recomputed live, not trusted from a snapshot. There is a denormalized PaymentMonth.expectedAmount column, but the read paths (student payment history, the outstanding summary) deliberately ignore it and re-sum the completed lessons every time, because that column goes stale the moment a lesson is later completed, cancelled, deleted, or re-priced. The stored snapshot exists mostly as a convenience written when you record a payment; the truth is always "sum the completed lessons' frozen rates."

Third: everything is bucketed by the teacher's timezone. The database stores every timestamp as UTC (@db.Timestamptz), but "which month does this lesson belong to?" is a question about the teacher's wall clock, not UTC. A lesson at 11:30pm on Jan 31 in Vietnam is a January lesson even though its UTC instant is February 1st. So month boundaries are computed in the teacher's IANA timezone and converted to UTC at the query edge.

As a React/SPA developer, the mental model shift is this: in a naive app you'd store student.rate and multiply rate × lessonCount on the fly. That's wrong here because it retroactively re-prices the past. Instead the money is immutable per lesson, and reporting is pure summation over frozen facts, grouped by the teacher's calendar. Once you internalize "freeze the fact, sum it live, bucket by local month," the rest of the code reads naturally.

The frozen per-lesson rate: why money lives on the Lesson, not the Student

The single most important business rule in PianoDiary: a lesson's price is a snapshot, captured once, and it never moves on its own. In the Prisma schema, Student has two current rate columns and Lesson has its own rate column:

  • Student.lessonRate — current in-person rate
  • Student.onlineLessonRate — current online rate
  • Lesson.isOnline — which of the two applies to this lesson
  • Lesson.rate — the frozen amount this specific lesson earned

The whole src/lib/rate.ts helper is just one tiny pure function, effectiveLessonRate, that picks the right current rate given isOnline. But the file's doc comment is where the policy lives — read it, because it's the contract the rest of the app upholds.

When a lesson is created (lesson.create), the router calls effectiveLessonRate(student, input.isOnline) and writes the result into Lesson.rate right then. From that point the lesson is a self-contained financial fact. If you later bump the student's lessonRate from 300 to 350, that create-time snapshot of 300 is untouched.

The why is history integrity / auditability. A piano teacher generates a report and a bill for February. If March's rate raise silently rewrote February's numbers, last month's already-sent invoice would no longer match the app. Freezing the rate onto each lesson makes the past immutable by construction — you literally cannot re-price it without running an UPDATE against those specific rows. That's the same instinct as storing the price on an order_line_item in e-commerce rather than joining live to the products table.

Analogy

Think of it like an e-commerce order line item: you copy the product price onto the order at checkout so that a later price change doesn't rewrite past receipts. Lesson.rate is that copied price.

Why

Storing money on the Student and multiplying live would be simpler to write but would retroactively re-price every past month whenever a rate changes — breaking already-issued reports and bills. Freezing per lesson makes the past immutable by construction.

Note

Rates are stored as Int — whole đồng, no decimals. There's no floating-point money anywhere, so sums are exact.

Re-pricing rules: when a frozen rate is allowed to change

"Frozen" doesn't mean "never changes" — it means "never changes by accident." There are exactly three deliberate re-stamp paths, each in a different router, and each is careful about scope.

1. Editing a student's rate (student.update) re-prices only the current month and future lessons, and only lessons still sitting at the old rate. It computes the start of the current month in the teacher's timezone, then runs updateMany with a where clause that includes rate: student.lessonRate (the OLD value). That extra rate filter is the clever bit: a lesson that was given a manual per-lesson override won't match, so it's left alone. Past months are never in the where, so they stay frozen.

2. Toggling a single lesson's online flag (lesson.update) re-stamps just that one lesson to the student's current applicable rate — even if the lesson is old. The reasoning (in the comment): flipping online/in-person is a deliberate correction to one lesson, unlike a bulk rate edit, so it's fine to re-derive its price now.

3. Marking attendance (lesson.markAttendance) has a priority order: an explicit per-lesson rate override always wins; otherwise, if you flip isOnline, it re-derives from the student's current rate.

So the frozen-ness is layered: bulk student-rate edits respect the past; per-lesson edits are allowed to move that one lesson; and an explicit override beats everything.

Gotcha

The rate: student.lessonRate filter in the student-update updateMany means a lesson you manually overrode to a custom price will NOT be swept up by a later bulk rate change. That's intentional — an override is a stronger signal than the student default.

Why

The month boundary uses getStartOfMonthUTC(now.getMonth()+1, ...) in the teacher's timezone, not new Date() truncated in UTC — otherwise a teacher in Vietnam editing a rate late at night could accidentally re-price (or fail to re-price) the boundary day.

Tip

createRecurring computes one recurringRate up front via effectiveLessonRate and stamps every generated lesson with it — so a whole recurring series is priced consistently at creation time.

The payments data model: PaymentMonth + PaymentTransaction, created lazily

Payments are modeled as two tables:

  • PaymentMonth — one row per (studentId, month, year) (there's a unique constraint). It carries a denormalized expectedAmount snapshot. Think of it as the folder for one student's one billing month.
  • PaymentTransaction — an individual receipt: an amount, an optional method and note, and a date. Many transactions hang off one PaymentMonth. A student paying in three installments creates three transactions in the same month folder.

Status is derived, never stored: UNPAID / PARTIAL / PAID, computed by derivePaymentStatus(expected, received).

The subtle design choice: PaymentMonth is created lazily. It is not created when you open the payments page or the "record payment" dialog. It's created only on the first actual transaction, via an upsert, inside addTransaction. So a student who owes money but hasn't paid yet has no PaymentMonth row at all — the payments table synthesizes a virtual row for them on the fly (with a fake id like ${studentId}-${month}-${year}) so the UI can still show "UNPAID, expected 1200."

Why lazy? Because eagerly creating a PaymentMonth for every student every month would litter the database with empty rows carrying a stale expectedAmount, and you'd have to keep them in sync as lessons change. Creating them only when money actually arrives keeps the table meaningful: a PaymentMonth row means "money has been recorded for this student-month."

Note

A student can PAID a month even if expected is 0 only in the sense of received >= expected. But watch the ordering: derivePaymentStatus(0, 0) returns UNPAID because received <= 0 is checked first. A zero-expected, zero-received month reads as UNPAID, not PAID.

Gotcha

Because PaymentMonth is lazy, the payments table's row id may be a synthetic string ${studentId}-${month}-${year} for students who haven't paid. Don't assume that id is a real database primary key — it becomes real only after the first transaction.

Why

Eagerly creating a PaymentMonth per student per month would create thousands of empty rows whose expectedAmount you'd have to keep synced as lessons change. Lazy creation means the row's existence itself carries meaning: 'money has been recorded here.'

The stale-snapshot problem: recompute expected LIVE from completed lessons

This is the most important correctness insight in the payments code, and it's worth reading the comment slowly.

PaymentMonth.expectedAmount is written only inside addTransaction — i.e. only at the moment a payment is recorded. But the expected amount is derived from the student's completed lessons that month, and lessons keep changing after that snapshot is taken: a PENDING lesson gets marked COMPLETE (expected goes up), a lesson gets CANCELLED or deleted (expected goes down), or a lesson gets re-priced. None of those events touch PaymentMonth.expectedAmount. So the stored snapshot goes stale and would disagree with the payments table.

The fix: the read paths that need to be trustworthy — getStudentHistory and getOverallSummary — ignore the stored snapshot and recompute expected live by summing the student's COMPLETE lessons, bucketed into months by the teacher's timezone. The stored column is essentially a cache that nobody fully trusts on read.

The pure helpers in src/lib/payment.ts do this bucketing and are unit-tested in isolation:

  • expectedByMonth(completedLessons, timezone) → a Map<"year-month", total>. It converts each lesson's UTC date to local time via fromUTC, keys by ${year}-${month}, and sums frozen rates. getStudentHistory uses this to attach a live expectedAmount to each historical month.
  • summarizeOutstanding(completedLessons, paymentMonths, timezone) → all-time { totalExpected, totalReceived, totalOutstanding } for the dashboard tile. The crucial detail: it buckets by student|year|month and clamps remaining per bucket before summing, so an overpaid month can't cancel out an underpaid one. The test proves it: Feb overpaid by 100, Mar underpaid by 100 → outstanding is 100, not 0.
  • calculateRemaining(expected, received) = Math.max(expected - received, 0) — never negative.
Why

Keeping a denormalized expectedAmount AND recomputing live looks redundant, but each has a job: the snapshot gives addTransaction a cheap value to store; the live recompute gives read paths a value that's always current. Reads never trust the cache.

Gotcha

If you ever add a feature that shows PaymentMonth.expectedAmount directly without recomputing, it will be subtly wrong for any month whose lessons changed after the last payment. Always route expected-amount reads through expectedByMonth / the live sum.

Tip

getForMonth and addTransaction's own return value DO compute expected from a fresh groupBy/aggregate of COMPLETE lessons in the month — so even the write path's response is live, not the just-stored snapshot.

Timezones: store UTC, convert at the edges, bucket months by the teacher's clock

Every timestamp column is @db.Timestamptz — stored in UTC. The teacher (User) has a timezone string (an IANA id like Asia/Ho_Chi_Minh, default UTC). src/lib/timezone.ts is the single place that converts between the two, built on date-fns-tz. The architecture is spelled out at the top of the file: DB stores UTC, user has a timezone preference, server converts between them.

The core primitives: - toUTC(date, tz) / fromUTC(utcDate, tz) — the two directions. fromUTC is used everywhere reporting code needs to ask 'what local day/month is this instant?' - createDateInTimezone(y, m, d, h, min, tz) — build a UTC instant from wall-clock parts interpreted in tz. Note month is 0-indexed here. - getStartOfMonthUTC(month, year, tz) / getEndOfMonthUTC(month, year, tz) — the month boundaries, as UTC instants, that bracket a calendar month in the teacher's zone. These are the workhorses: nearly every query that filters 'lessons in month X' uses date: { gte: startOfMonthUTC, lte: endOfMonthUTC }.

Why store-UTC-convert-on-read? UTC is the one unambiguous instant; a raw wall-clock time is meaningless without a zone and breaks across DST. So the database holds instants, and interpretation happens only when a human's calendar matters. The classic bug this prevents: a lesson at 11:30pm Jan 31 in Vietnam (UTC+7) has a UTC instant of Feb 1 00:30. If you bucketed by UTC month it would land in February and vanish from January's report and January's bill. getStartOfMonthUTC(1, ...) / getEndOfMonthUTC(1, ...) in Asia/Ho_Chi_Minh produce boundaries that correctly capture it as a January lesson.

getEndOfMonthUTC is implemented as 'start of next month minus 1ms', which sidesteps leap-year and 30-vs-31-day edge cases. And formatInTimezone carries a sharp comment: date-fns-tz's format does not shift the wall clock on its own, so the code converts the instant into the zone first, then formats.

These helpers are unit-tested with zone-explicit assertions, plus a dedicated test:tz npm script that forces TZ=America/New_York to catch code that accidentally depends on the host machine's timezone — because, per the test comment, mutating process.env.TZ at runtime is a no-op on Windows.

Gotcha

createDateInTimezone takes a 0-indexed month (0 = January) to match JS Date, but getStartOfMonthUTC / getEndOfMonthUTC take a 1-indexed month (1 = January). Mixing them up is an easy off-by-one — the boundary helpers convert internally with month - 1.

Why

Store-UTC-convert-on-read means the DB holds one canonical instant and 'what month/day is this?' is answered only where a human calendar matters. It survives DST transitions and lets the same lesson be correct for viewers in different zones.

Tip

Run npm run test:tz (it sets TZ=America/New_York) before trusting any new date logic — it surfaces code that accidentally reads the host machine's timezone instead of the teacher's. On Windows you can't flip process.env.TZ at runtime, so this separate script exists to launch the process under a forced zone.

Note

The teacher's timezone is read from ctx.session.user.timezone ?? "UTC" at the top of nearly every router procedure. UTC is the safe default when a user hasn't picked one.

Worked example: completing a lesson → expected → recording a payment → status

Let's trace one full cycle for a student in Vietnam (Asia/Ho_Chi_Minh, UTC+7) whose current lessonRate is 300.

1. Create the lesson. The teacher schedules an in-person lesson for Feb 10. lesson.create runs effectiveLessonRate(student, false) → 300, and writes Lesson.rate = 300, status = "PENDING". That 300 is now frozen onto this row.

2. The student raises to 350 later. student.update re-stamps only current-and-future lessons still at the old rate. If Feb is already past, the Feb 10 lesson keeps 300. History intact.

3. Mark attendance. lesson.markAttendance sets status = "COMPLETE". The lesson now counts toward expected. Its rate is still the frozen 300 (no override, no online flip).

4. Expected for February. Nobody stores this eagerly. When the payments page loads (getForMonth), buildMonthPaymentRows computes February's boundaries in Vietnam time (getStartOfMonthUTC(2, 2026, tz) = Jan 31 17:00 UTC), does a groupBy sum of rate over COMPLETE lessons in that window, and gets expectedAmount = 300. With no payment yet, the row is synthetic: receivedAmount = 0, remaining = 300, status = UNPAID.

5. Record a payment of 200. addTransaction aggregates COMPLETE-lesson rates for the month (300), upserts the PaymentMonth (creating it now, storing expectedAmount = 300), then creates a PaymentTransaction for 200. The payment date string is parsed to a UTC instant via createDateInTimezone so it lands on the right calendar day. It returns received = 200, remaining = calculateRemaining(300, 200) = 100, status = derivePaymentStatus(300, 200) = PARTIAL.

6. Read it back in history. getStudentHistory does NOT trust the stored expectedAmount = 300. It re-sums COMPLETE lessons via expectedByMonth → 300, recomputes received = 200, remaining = 100, status = PARTIAL. If later a second Feb lesson (rate 300) is marked COMPLETE, the live recompute jumps expected to 600 and the month flips back toward UNPAID/PARTIAL correctly — even though the stored snapshot still says 300.

Note

A payment date arrives as a YYYY-MM-DD string and is turned into a UTC instant with createDateInTimezone(...,0,0,tz) via parseDateOnlyToUTC, so a payment recorded on the 1st in Vietnam doesn't slip into the previous month.

Analogy

Received is like summing the deposits in a bank statement; expected is recomputed from the invoice each time you look. Status is just comparing the two — never a value you have to remember to update.

Reports: MonthlyReport with a JSON metadata snapshot, unique per student/month/year

The report router produces the monthly write-up a teacher hands to a student/parent. There are two flavors of data:

  • Live computed — generatePreview and getStudentReport pull the month's lessons (filtered by timezone-aware boundaries) and sum each lesson's frozen rate into totalFee. So the money on a report is exactly the frozen-rate money everywhere else. It also returns the student's current lessonRate/onlineLessonRate for display.
  • Persisted narrative — MonthlyReport stores the teacher's prose (summary, comments, nextMonthPlan, tuitionNote) plus a lessonMetadata JSON blob. That JSON is a per-lesson annotation map (keyed by lesson id → { reason, customReason, isSpecial, color }) — e.g. tagging why a particular lesson was special or how to color it in the printed report.

MonthlyReport has a @@unique([studentId, month, year]) constraint, so there's exactly one report per student per month, and upsertReport uses that composite key to create-or-update. The lessonMetadata field is validated with a Zod z.record capped at 200 entries and bounded string lengths — a guard against a malformed or oversized JSON blob reaching the database.

Why store metadata as JSON rather than a proper table? Because it's free-form presentation state attached to a report, never queried or aggregated across reports — it's only ever read back whole for the one report it belongs to. A JSON column is the pragmatic choice for 'a bag of per-lesson annotations I'll render as-is.'

Note

getStudentReport reads the timezone from teacher.user.timezone (it includes user), whereas most payment/earnings procedures read ctx.session.user.timezone. Same value in practice, different access path — don't be surprised by the variation.

Why

lessonMetadata is a Prisma Json column (@default("{}")) rather than a relational table because it's opaque render state read back whole for a single report — never filtered or aggregated. A table would add joins for zero query benefit.

Earnings & dashboard intelligence: aggregation over frozen rates, bucketed by local month

The earnings router powers the dashboard tiles. Every number it produces is a sum over the same frozen Lesson.rate, sliced by timezone-aware month boundaries and by status. A few representative procedures:

  • getDashboard — headline tiles: totalEarnings (all COMPLETE lessons, all time), currentMonthEarnings (COMPLETE this month), currentMonthLoss (CANCELLED this month — the money lost to cancellations, since cancelled lessons still carry their frozen rate), plus last-month collected/outstanding by re-summing expected-vs-received per student.
  • getTopStudentsThisMonth — groups this month's COMPLETE lessons by student, ranks by lesson count (not earnings), tie-broken alphabetically for stable ordering, returns the top N (default 5). This feeds the 'Top Students This Month' card.
  • getEarningsTrendThisMonth — builds one point per day of the month, sums each COMPLETE lesson's rate into its local day bucket (fromUTC(...).getDate()), and trims to days up to today. This is the earnings line chart.
  • getQuickInsights — completed/cancelled counts, a completionRate, the teacher's bestDay (weekday with the most completed lessons, computed in local time), and an inactiveCount: students with no COMPLETE lesson in the last 14 days (the 'follow-up list').

On the client, DashboardIntelligencePanel (a Client Component — note "use client") fires these as separate tRPC queries and lays them out. DashboardTopStudentsCard renders the ranking with medals for the top 3; DashboardQuickInsightsCard shows best day / taught / cancelled / follow-up list.

Separately, BirthdayCountdownCard is not domain data at all — it's a hardcoded countdown to 2026-04-24 with a client-side easter egg (click 100+ times to reach /birthday-room). Worth knowing it exists so you don't go hunting for a 'birthday' table; there isn't one.

Gotcha

'Top students' ranks by lesson COUNT, not earnings, even though the card is on an earnings-flavored panel and the row type carries an earnings field. Don't assume the medal order tracks money — a cheap-rate student with many lessons outranks an expensive one with few.

Note

getQuickInsights's 'recent' window uses a raw rolling new Date(Date.now() - 14 days) rather than a timezone month boundary — it's a 14-day lookback, not a calendar-month slice, so timezone bucketing isn't needed there.

Analogy

These procedures are just GROUP BY reports you'd write in SQL, but done in TS after a Prisma fetch — group the frozen facts, bucket by the teacher's calendar, sum. Same shape as the payments helpers, different slice.

Key takeaways

  • Money is frozen per lesson: Lesson.rate is snapshotted from the student's current in-person/online rate at creation time and never moves on its own — that's what keeps past months' reports and bills immutable.
  • There are exactly three deliberate re-price paths: a student rate change (current+future only, and only rows still at the old rate), toggling one lesson's online flag (that single lesson), and an explicit per-lesson override (which beats everything).
  • PaymentMonth is created lazily via upsert on the first transaction — not on page load — so its existence means 'money was recorded here.' Students who owe but haven't paid get a synthetic virtual row.

  • All timestamps are stored UTC (@db.Timestamptz); month boundaries are computed in the teacher's IANA timezone via getStartOfMonthUTC/getEndOfMonthUTC so a late-night lesson lands in the correct local month, not a UTC-shifted one.
  • Payment status (UNPAID/PARTIAL/PAID) is always derived from expected-vs-received, never stored — and derivePaymentStatus checks received <= 0 first, so a 0-expected/0-received month reads UNPAID.

Tailwind v4's CSS-first setup, the shadcn/Radix 'own your components' pattern, design tokens & theming, charts, tables, and a little whimsy.

If you're coming from plain React, you already know the hard part: components, props, hooks, and JSX. So "the component library" is really "Radix behavior + our styles, owned in our repo."

Tailwind CSS v4: the config-light, CSS-first setup

In older Tailwind (v3), you configured everything in a tailwind.config.js file — your color palette, spacing scale, plugins — all in JavaScript. Tailwind v4 moved configuration into CSS. This repo has no tailwind.config.js at all (I checked — the glob finds nothing). Instead, the whole thing is bootstrapped from three places.

1. The entry point is src/styles/globals.css, which starts with @import "tailwindcss";. That one line pulls in all of Tailwind's base styles, utility generation, and the new engine. The next line, @import "tw-animate-css";, adds a small library of animation utilities (the animate-in, fade-in-0, zoom-in-95 classes you'll see on dialogs).

2. The build step is PostCSS. postcss.config.js is a five-line file whose only job is to register the @tailwindcss/postcss plugin. In v4, that single plugin is Tailwind — it scans your files for class names and generates the CSS. There's no separate tailwindcss CLI config to think about.

3. The design tokens live in an @theme { ... } block inside globals.css. Anything you define there becomes both a CSS variable and a Tailwind utility. For example, defining --color-primary makes bg-primary, text-primary, border-primary all work automatically. This is why you see custom radii like rounded-4xl used in the app — they're declared in @theme (--radius-4xl), not built in.

The @custom-variant dark (&:is(.dark *)) line at the top defines how the dark: prefix behaves: a dark: utility applies whenever the element is inside something with the .dark class. That's the hook the whole dark-mode system hangs on (more on that in the theming section).

globals.css (@import tailwindcss + @theme tokens)
        │
   PostCSS  ──▶  @tailwindcss/postcss plugin scans class names
        │
   generated CSS (utilities + CSS variables) ──▶ browser
Gotcha

Don't look for tailwind.config.js — it doesn't exist in this repo. To add a color, radius, or animation, edit the @theme block in src/styles/globals.css instead.

Why

CSS-first config means your design tokens are real CSS variables at runtime. That's what makes runtime theming (dark mode, birthday mode, per-chart colors) possible without rebuilding — you just change the variable's value.

Note

@theme tokens follow a naming convention: --color- generates color utilities, --radius- generates rounded- utilities, --animate- generates animation utilities. The custom --animate-piano keyframe in this file powers the app's loading indicator.

Note

globals.css actually has two token blocks: a plain @theme { --font-sans: … } and a separate @theme inline { … } holding the radii, all --color-* mappings, and --animate-piano. The inline keyword makes generated utilities emit var(--primary) rather than baking the value at build time — that indirection is exactly what lets :root / .dark swap values at runtime. A @layer base rule (* { border-border }) also gives every element a default token border, which is why borders 'just work'.

Why it's built this way

The T3 / shadcn starter this project is based on adopted Tailwind v4 early. The tradeoff: less boilerplate and runtime-swappable tokens, at the cost of the config being somewhere a v3 veteran won't expect (in CSS, not JS).

The cn() helper: why merging classes matters

This is a five-line file you will import in almost every component, so it's worth understanding deeply. cn() does two things by composing two libraries:

  1. clsx turns a messy list of arguments — strings, conditionals, arrays, objects like { 'items-center': isDot } — into one flat className string. This is the ergonomic part: you can write cn('base', isActive && 'text-primary', className) and falsy values just drop out.
  1. tailwind-merge (twMerge) is the clever part. Plain string concatenation has a bug with Tailwind: if you write class="px-2 px-4", both classes are in the DOM, and which one wins is decided by CSS source order — not by which you wrote last. So a component with a default px-2 can't be reliably overridden by a caller passing px-4. twMerge understands Tailwind's class groups: it knows px-2 and px-4 are the same 'padding-x' property and keeps only the last one. The caller's override actually wins.

That's the whole reason every UI component ends with className={cn(someDefaults, className)} — it lets a parent pass className and reliably override the component's built-in styles. Without twMerge, passing className="rounded-full" to a button that defaults to rounded-md might or might not work depending on stylesheet order. With it, it always works.

Analogy

Think of cn() as classNames() (the classic React lib) with a Tailwind-aware conflict resolver bolted on. clsx is the classNames part; twMerge is the new superpower.

Tip

Rule of thumb: put the caller's className LAST in the cn() call. twMerge resolves conflicts left-to-right, so last-wins is what lets overrides take effect.

Gotcha

twMerge only understands standard Tailwind class patterns. Arbitrary or non-Tailwind classes it can't categorize are simply kept as-is (not merged), so two conflicting custom classes can still both appear.

class-variance-authority (cva): giving components typed variants

How does <Button variant="destructive" size="lg"> map to a specific set of Tailwind classes? That's class-variance-authority (cva). You give cva() a base string of classes that always apply, then a variants object that maps prop values to class strings, plus defaultVariants for when the prop is omitted. cva returns a function; call it with { variant, size } and it returns the merged class string.

The pairing with TypeScript is the nice part: VariantProps<typeof buttonVariants> extracts the exact union types ('default' | 'destructive' | 'outline' | ...) so the component's props are fully typed and autocompleted — pass an invalid variant and TS complains.

Two project-specific conventions to notice in button.tsx:

  • asChild + Radix Slot. When asChild is true, the component renders Slot instead of <button>. Slot merges the Button's props and classes onto its child element instead of rendering a button. This is how you make a Next.js <Link> look like a button: <Button asChild><Link href=…>Go</Link></Button> yields a single styled <a>, not a button wrapping a link (which would be invalid HTML). You'll see this pattern all over the codebase.
  • data-slot / data-variant / data-size attributes. Every component stamps a data-slot attribute. These aren't for JS — they're styling hooks. The birthday-mode CSS in globals.css, for instance, targets button[data-slot="button"]:not([data-variant="ghost"])… to add a gradient hover only to solid buttons. It's a clean way to reach into components from global CSS without prop drilling.
Note

cva accepts className inside its argument object (see buttonVariants({ variant, size, className })). It threads the caller's className through so twMerge can resolve conflicts against the variant classes.

Tip

Want to see another cva example? src/components/ui/badge.tsx is a smaller one — same shape, just a variant axis (default/secondary/destructive/outline) and no size.

Analogy

cva is to className what a styled-component's prop-based styling is — but it stays as plain utility strings, so there's no runtime CSS-in-JS cost.

The shadcn/ui pattern: owned components on top of headless Radix primitives

This is the mental model shift. In a typical React app you'd npm install some-ui-lib and import { Dialog } from 'some-ui-lib'. shadcn/ui is different: it's a catalog of components you copy into your own repo. Everything under src/components/ui/* (there are ~35 files: button, dialog, sheet, select, tabs, popover, dropdown-menu, switch, tooltip, sidebar, table, calendar, and more) is source you own. There's no shadcn package in package.json. If a dialog needs different padding, you edit dialog.tsx directly.

What these files wrap is Radix UI primitives (the @radix-ui/react-* packages you do see in package.json). Radix primitives are headless: they ship all the hard behavior — focus trapping, keyboard navigation, ARIA attributes, portal rendering, open/close state — but zero visual styling. shadcn's job is to be a thin styling layer: re-export each Radix part as a small local function that adds Tailwind classes via cn() and a data-slot attribute.

Look at dialog.tsx. Dialog, DialogTrigger, DialogPortal, DialogClose are near-passthroughs to DialogPrimitive.Root, .Trigger, etc. The only real work is in DialogContent and DialogOverlay, which add the positioning, the semi-transparent backdrop, the enter/exit animation classes (data-[state=open]:animate-in, data-[state=closed]:fade-out-0 — those come from tw-animate-css), and a close button. Radix handles that the dialog opens, traps focus, closes on Escape, and renders in a portal; this file handles what it looks like.

The data-[state=open]: / data-[state=closed]: selectors are worth calling out: Radix sets a data-state attribute on its elements as they open and close, and the styling reacts to it with pure CSS. No animation library needed for these — it's Tailwind utilities keyed off Radix's state attribute.

Why

Owning the component code (vs. installing a library) means no fighting a library's theming API and no waiting for upstream fixes — but it also means updates are manual: you re-copy from shadcn when you want new versions. This project has clearly customized several of these files.

Note

'Headless' = behavior without looks. Radix guarantees accessibility and interaction correctness; you supply 100% of the CSS. This is the same philosophy behind @tanstack/react-table (headless tables) used later.

Gotcha

Most of these files start with "use client". Radix primitives use state and effects, so they can't be React Server Components. Any file importing them becomes a client component too.

Tip

The data-slot convention is used project-wide as a styling anchor. Birthday mode reaches components via [data-slot="card"] and button[data-slot="button"] selectors in globals.css — a nice example of styling components you don't want to edit individually.

Theming: design tokens, dark mode, and an honest wiring caveat

The color system is entirely CSS variables. :root in globals.css defines the light palette — --background, --foreground, --primary, --card, --border, five --chart-* colors, and a full --sidebar-* set — all in the modern oklch() color space (perceptually uniform lightness, nicer for generating hover tints). A .dark { ... } block re-declares every one of those same variables with dark values. Because the @theme inline block maps --color-primary: var(--primary) and friends, the Tailwind utilities (bg-primary, text-foreground) always read the current value of the variable. So when the .dark class is present on an ancestor, every token flips at once and the whole UI re-themes — no per-component dark styling needed. That's the payoff of CSS-first tokens.

The intended mechanism to add/remove that .dark class is next-themes, the standard Next.js dark-mode library. It's installed, and one component consumes it: sonner.tsx (the toast component) calls useTheme() so toasts match the theme.

Here's the honest caveat you need to know: I searched the whole repo and there is no ThemeProvider mounted anywhere (grep for ThemeProvider returns nothing; the root Providers component wires SessionProvider, tRPC, UserStore, and Birthday — but not next-themes). Without a ThemeProvider, next-themes' useTheme() returns its default and nothing ever toggles the .dark class. So in practice the app currently runs in light mode; the dark tokens are fully defined and ready, but the switch to activate them isn't plugged in. (The project docs describe dark mode as 'wired up,' but the code doesn't yet mount the provider — treat the docs as aspirational here.) If you're asked to 'turn on dark mode,' the task is to add next-themes' ThemeProvider in src/components/providers/index.tsx and a toggle — not to write new CSS, which already exists.

Separately, there's a second, custom theme layer: 'birthday mode' toggles a .birthday-mode class on <html> (set imperatively in birthday-provider.tsx), and a large block of scoped CSS in globals.css restyles cards, buttons, and inputs only while that class is present. It's the same technique as .dark — a class on a root element flips a bunch of scoped rules — just hand-rolled for a feature.

Gotcha

Dark mode is not currently active in the running app: the .dark CSS tokens exist and next-themes is installed, but no ThemeProvider is mounted to toggle the .dark class. Don't assume dark: utilities are exercised at runtime yet.

Why

Defining light and dark as the SAME set of CSS variable names is what makes theming a one-class-flip. Components never reference a color literal — only tokens — so they're theme-agnostic by construction.

Note

Colors use oklch() rather than hex. oklch separates lightness from hue, so bg-primary/90 (a 90%-opacity tint used for hovers) stays perceptually consistent across the palette.

Analogy

Birthday mode is 'dark mode' built by hand: instead of next-themes toggling .dark, the BirthdayProvider toggles .birthday-mode on <html>, and scoped CSS in globals.css does the reskin.

Fonts via next/font, and the app's Providers tree

Fonts are loaded with next/font, Next.js's built-in font optimizer. In src/app/layout.tsx, Plus_Jakarta_Sans is imported from next/font/google; calling it at module scope downloads and self-hosts the font at build time (no runtime request to Google, no layout-shift flash) and returns an object with a .variable class. That class is applied to <body>, which sets the CSS variable --font-plus-jakarta-sans. Only weights 600 and 700 are requested — this font is used for bold headings; body text falls back to the font-sans stack.

One small inconsistency worth knowing so it doesn't confuse you: the @theme block sets --font-sans to start with var(--font-geist-sans), but that variable isn't defined anywhere in this repo (it's a leftover from the T3/shadcn starter, which used Geist). Since it's undefined, the browser skips it and falls through to ui-sans-serif, system-ui, …. Plus Jakarta is loaded and available via its own variable, but the default font-sans token doesn't point at it. Not a bug that breaks anything — just don't assume font-sans = Plus Jakarta.

The layout.tsx also shows the App Router root: it renders <html> and <body> (something you don't do in a plain SPA) and wraps children in <Providers>. That Providers component (src/components/providers/index.tsx) is the client-side context stack: an ErrorBoundary, then NextAuth's SessionProvider, then TRPCReactProvider, then a UserStoreProvider (Zustand), and conditionally a BirthdayProvider, plus the Toaster. This is the standard Next.js split: the layout is a Server Component (can export metadata), and all the interactive context providers live in a single "use client" component it renders.

Why

next/font downloads fonts at build time and serves them from your own domain, eliminating the render-blocking request to Google Fonts and the font-swap flicker. You get a .variable class instead of a global font — so you opt fonts in per-element via CSS variables.

Gotcha

--font-sans in @theme references var(--font-geist-sans), which is undefined in this repo (a starter leftover). It harmlessly falls back to the system sans stack. If you want Plus Jakarta as the default body font, point --font-sans at var(--font-plus-jakarta-sans).

Note

suppressHydrationWarning on <html> is standard when a theme/class may be set on the client before hydration — it silences the expected server/client mismatch on that element.

Charts with recharts, wrapped by a local chart.tsx

Charts use recharts (a declarative React charting library — you compose <LineChart>, <XAxis>, <Line> as components). Rather than use recharts raw, this repo copied shadcn's chart.tsx wrapper into src/components/ui/. It provides four things worth understanding:

  1. ChartContainer — wraps recharts' ResponsiveContainer, provides a React context holding your config, and (via ChartStyle) injects a <style> tag that turns each config entry's color into a CSS variable like --color-earned. That's the trick that lets you write stroke="var(--color-earned)" and have it theme-aware and defined in one place. ChartStyle even emits separate rules for light and .dark when a config entry provides a theme: { light, dark } object.
  1. ChartConfig — a typed object mapping each data series to a label, optional icon, and a color (or per-theme colors). It's the single source of truth a chart's tooltip and legend read from.
  1. ChartTooltipContent and ChartLegendContent — pre-styled, theme-aware replacements for recharts' defaults, wired to read labels/colors from the config context.

A real usage is the dashboard's earnings trend card (dashboard-earnings-trend-card.tsx). It declares a chartConfig with satisfies ChartConfig (which type-checks the shape without widening it), then renders a <LineChart> inside <ChartContainer> with a custom SVG gradient stroke, a themed ChartTooltip, and a ChartLegend. Note the loading/empty states are handled around the chart with a Skeleton and a fallback message — a good pattern to copy. The five --chart-1..5 tokens from globals.css are the intended default series palette.

Note

satisfies ChartConfig validates the object against the type while keeping its literal keys (so chartConfig.earned still autocompletes). Prefer it over : ChartConfig for config objects.

Tip

ChartContainer must wrap a single recharts chart element as its child — its children type is recharts' ResponsiveContainer child. Loading and empty states go outside/around it, as the earnings card demonstrates.

Why

Injecting colors as CSS variables (rather than hardcoding hex in JSX) keeps chart series colors themeable and centralized in one config object that the chart, its tooltip, and its legend all share.

Gotcha

Heads-up on the ChartStyle → var(--color-<key>) mechanism: the one real example (dashboard-earnings-trend-card) doesn't actually consume it — its line is painted with a hardcoded SVG gradient (url(#dashboard-earnings-gradient)) and pink dots. The injected CSS-variable approach is real, but this chart opted out of it.

Note

recharts here is v2 (^2.15.4), not v3. v3 changed several APIs, so if you copy newer shadcn chart.tsx snippets or recharts docs, pin your mental model to v2 patterns.

Tables with @tanstack/react-table behind a shared DataTable

Tabular data uses @tanstack/react-table — another headless library (like Radix, it gives you logic — sorting, filtering, pagination, row models — and no markup). This repo adds a shared src/components/ui/data-table.tsx that renders the actual <Table> markup (itself a thin shadcn wrapper over <table> elements) and, cleverly, supports two modes via a discriminated union:

  • Tanstack mode — you pass a fully-built table object from useReactTable(...). DataTable then renders it with flexRender and getHeaderGroups()/getRowModel(). This is used where you need sorting/filtering/pagination — e.g. students-table.tsx and pieces-table.tsx, which call useReactTable with getCoreRowModel, getSortedRowModel, getFilteredRowModel, and getPaginationRowModel, and pair it with the shared DataTablePagination footer.
  • Basic mode — you pass plain columns (each with an id, header, and a cell(row) render function) and data. No sorting/pagination machinery; just a clean styled table. This is used by payments-table.tsx, which defines its columns inline and also renders a mobile card layout as a fallback below md. The runtime tells the modes apart with an "table" in props type guard (isTanstackDataTable).

Also in this file: DataTableState (styled empty/loading placeholder), DataTableToolbar, and DataTableViewToggle (a grid/table segmented switch). The payments table shows the basic mode plus a nice responsive pattern — real <DataTable> on desktop (hidden md:block), hand-built cards on mobile (md:hidden).

Tip

Choosing a mode: if the table just displays rows, use basic mode (inline columns/data) like payments-table. If you need sorting/filtering/pagination, build a useReactTable instance and pass table={...} — then you also get DataTablePagination for free.

Why

One DataTable with two shapes avoids duplicating the styled <table> markup (sticky header, rounded pink border, scroll container) across every list screen while still allowing the full power of tanstack where it's warranted.

Note

@tanstack/react-table is headless like Radix: flexRender + the row-model getters are logic only. All the visual chrome lives in the shadcn table.tsx primitives that DataTable composes.

Tooling, env & testing

The guardrails: type-safe env vars, the npm scripts you'll run daily, TypeScript paths, linting, and the timezone-aware test suite.

This subsystem is the "guardrails layer" of the app — the stuff that isn't a feature you can click, but that keeps every feature honest: it makes sure environment variables exist and have the right shape before the app even boots, gives you one-word commands for every dev task, wires up TypeScript path aliases so imports stay clean, and runs a test + lint + typecheck gate on every push. If you come from a plain React/Vite SPA, most of this will feel familiar in spirit (you've seen .env, ESLint, maybe Jest) but the details are tuned for the T3 stack (Next.js + tRPC + Prisma + Zod), and a few pieces exist specifically because Next.js runs your code in two places at once — on the server and in the browser.

The single most important idea here is fail fast on config. In a normal SPA, if you forget to set DATABASE_URL, you find out at 2 a.m. when a request quietly reads process.env.DATABASE_URL as undefined and hands it to a database driver that throws a cryptic error. This project instead validates every environment variable against a Zod schema in src/env.js at build and boot time. A missing or malformed variable crashes the build with a clear message naming the variable — long before a user ever hits it. That file is imported at the top of next.config.js, so next build and next dev literally cannot start with a broken config.

The second big idea is the server/client boundary, which shows up over and over in a Next.js codebase. Server code can read secrets (DATABASE_URL, AUTH_SECRET); browser code must never see them. @t3-oss/env-nextjs enforces this split: server variables and client variables (which must be prefixed NEXT_PUBLIC_) are declared in separate blocks, and if you try to read a server-only variable from client code, you get an error instead of a leaked secret. The server-only npm package is a related guard: importing it at the top of a module makes the build fail if that module is ever bundled into the browser.

The rest is developer ergonomics: package.json scripts wrap Next, Prisma, ESLint, Prettier, and Vitest into memorable verbs; tsconfig.json sets up strict TypeScript plus the @/* alias so you write @/lib/timezone instead of ../../../lib/timezone; Vitest (a Vite-native test runner, basically a faster Jest) runs the pure-logic tests with a superjson transformer and a special timezone test mode; and a small GitHub Actions workflow runs lint + typecheck + tests on every push to main. None of this is business logic, but it's the scaffolding that lets the business logic be changed safely.

Type-safe environment variables (src/env.js)

In a plain React app you read import.meta.env.VITE_FOO or process.env.FOO directly and just hope it's set. This project instead funnels every environment variable through one file, src/env.js, which uses @t3-oss/env-nextjs (a thin T3 wrapper around Zod) to declare a schema and validate process.env against it.

The call to createEnv has three important blocks:

  • server — variables only backend code may read. DATABASE_URL must be a valid URL (z.string().url()). AUTH_SECRET is required in production but optional in development — notice the ternary keyed on NODE_ENV, so a fresh clone can run dev without a secret but a production build refuses to start without one. NODE_ENV itself is a Zod enum with a default of "development".
  • client — variables safe to ship to the browser. The library forces these to start with NEXT_PUBLIC_; that prefix is Next.js's own convention for "OK to inline into the client bundle." Here there's just NEXT_PUBLIC_ERROR_TRACKING_URL, an optional error sink.
  • runtimeEnv — a manual mapping of each key back to process.env.X. This looks redundant but is required: in Next.js's edge runtime and in client bundles you cannot dynamically iterate process.env, so each variable must be named explicitly for the bundler to statically inline it.

After this, the rest of the app imports env from @/env and reads e.g. env.DATABASE_URL — now a typed, guaranteed-present value, not a string | undefined. That's the payoff: no more ! non-null assertions or defensive if (!process.env.X) throw scattered everywhere.

One accuracy note for the handbook: the topic brief mentioned GOOGLE_CLIENT_ID/SECRET, but this repo's env.js does not declare those — its real server vars are AUTH_SECRET, DATABASE_URL, CLOUDINARY_URL, and NODE_ENV. Don't document env vars that aren't there.

Why

The whole point is fail-fast: a bad or missing env var crashes the build with a message naming the variable, instead of surfacing as an undefined value deep in a request at runtime in production.

Gotcha

Every variable must appear in BOTH its schema block AND runtimeEnv. Forgetting the runtimeEnv entry means the value reads as undefined at runtime even though the schema looks right — a classic newcomer trap in this file.

Analogy

Think of env.js as PropTypes/Zod, but for your deployment environment instead of a component's props: it validates the 'inputs' your app receives from the outside world once, at the boundary.

Why it's built this way

T3 chose runtime Zod validation over the simpler 'just read process.env' because config errors are the most expensive kind to discover late — they pass code review, pass tests that mock the env, and only bite in the target environment. Validating at build/boot moves that failure left to the earliest possible moment.

skipValidation and how env.js gets wired into the build

Two flags at the bottom of createEnv matter. emptyStringAsUndefined: true means an env var set to an empty string (FOO="") is treated as unset — so a required z.string() will correctly reject it instead of accepting a useless empty value. skipValidation: !!process.env.SKIP_ENV_VALIDATION provides an escape hatch: when that variable is present, the whole schema check is bypassed.

Why would you ever skip the very validation that's the point of the file? Two legitimate cases: (1) Docker/CI image builds where you compile the app without real secrets present, and (2) unit tests, where you don't want importing a module that transitively pulls in env.js to explode just because the test runner has no DATABASE_URL. You'll see this exact flag set in vitest.config.ts (next section).

The enforcement mechanism is the first line of next.config.js: it does import "./src/env.js"; before exporting any config. Because Next.js loads its config at the very start of build and dev, that import runs the validation immediately. If a var is missing, the process dies before Next does any real work. This is the 'no undefined env in prod' guarantee made concrete — there's no code path into the running app that hasn't passed through this check.

Tip

If your npm run build fails with a Zod error listing env vars, you don't have a code bug — you have a missing .env entry. Copy .env.example to .env and fill it in.

Note

SKIP_ENV_VALIDATION disables the schema entirely, so env.DATABASE_URL becomes whatever raw string (or undefined) is in the environment. That's fine for tests that mock the DB, but never set it in real deployments.

Why it's built this way

An escape hatch is necessary precisely because the validation is so strict — without skipValidation, you couldn't build a Docker image or run a pure unit test without provisioning every production secret. The flag keeps the strict default while allowing the few environments where strictness is counterproductive.

The package.json scripts — one verb per task

package.json is the command palette for the repo. Here's what each script actually does and, importantly, the ones that are easy to confuse.

Next.js lifecycle: - dev → next dev — hot-reloading dev server. - build → next build — production compile (runs env validation). - start → next start — serve an already-built app. - preview → next build && next start — build then serve, i.e. a local dry-run of production in one command.

Prisma (three DB commands that look alike but are NOT interchangeable): - db:push → prisma db push — shoves the current schema straight into the database with no migration file. This is the one this project actually uses (per the repo's own memory note: the Neon Postgres migration history is out of sync, so migrate deploy must never be run). - db:generate → prisma migrate dev — creates a new migration file and regenerates the client (dev-only, interactive). - db:migrate → prisma migrate deploy — applies committed migrations in production. Named db:migrate but effectively off-limits here. - db:studio → prisma studio — a local GUI to browse/edit rows.

TypeScript config and the @/* path alias

tsconfig.json turns on strict, modern TypeScript. The strictness knobs worth knowing: strict: true (the usual bundle of strict checks), noUncheckedIndexedAccess: true (indexing an array/object gives T | undefined, forcing you to handle the missing case — stricter than most projects), and checkJs: true with allowJs: true (so plain .js files like env.js and the config files are also type-checked). verbatimModuleSyntax and isolatedModules are there because a bundler (Next/Vite), not tsc, actually transpiles the code — noEmit: true means tsc is used purely as a type checker, never to produce output.

The part you'll touch daily is path aliases. Instead of relative-path spaghetti like ../../../lib/timezone, this project maps @/* to ./src/*, so you import @/lib/timezone. There's a second alias, @prisma/* → ./generated/prisma/*, pointing at the generated Prisma client location.

Here's the Next.js/newcomer catch: an alias in tsconfig.json only teaches the TypeScript language server and Next's bundler about the mapping. Other tools don't automatically read it. Vitest, in particular, runs through Vite and would not understand @/lib/... on its own. That's why the project adds the vite-tsconfig-paths plugin in vitest.config.ts — it reads the same tsconfig.json paths so your test imports resolve identically to your app imports. One source of truth, honored everywhere.

Gotcha

A tsconfig path alias is understood by tsc/your editor and Next's bundler — but NOT automatically by Vitest, Jest, or Node at runtime. Each tool needs its own way to learn the mapping; here it's the vite-tsconfig-paths plugin.

Note

noUncheckedIndexedAccess is stricter than most codebases: arr[0] is typed T | undefined. Expect to handle 'possibly undefined' on array/object indexing — that's intentional, not a misconfiguration.

Note

tsconfig.json's exclude lists "generated", so the generated Prisma client at ./generated/prisma/* is left out of the project's own typecheck even though the @prisma/* alias points there.

Why it's built this way

Pointing the Vitest plugin at the same tsconfig.json (rather than duplicating aliases in the Vitest config) keeps a single source of truth — add an alias once and both the app and the tests pick it up, so tests can never drift from how the app resolves modules.

ESLint (flat config) and Prettier

ESLint here uses the modern flat config format (eslint.config.js exporting an array), not the legacy .eslintrc. It's composed from eslint-config-next/core-web-vitals (Next's recommended rules, including accessibility and Core Web Vitals hints) plus typescript-eslint's three tiers: recommended, recommendedTypeChecked, and stylisticTypeChecked. The 'TypeChecked' variants are powerful because they use real type information (not just the syntax tree) to catch bugs like floating promises — which is why languageOptions.parserOptions.projectService: true is set at the bottom to give ESLint access to the TypeScript project.

The custom rules block tunes a few defaults: consistent-type-imports nudges you toward import type, unused vars are a warning (and vars prefixed _ are ignored), and no-misused-promises is relaxed for JSX attributes so you can pass an async function to an event handler. There's also a deliberate downgrade: ESLint 16 bundles React-Hooks 7, whose new 'React Compiler readiness' rules (set-state-in-effect, purity) would flag lots of existing, working code as errors — the config keeps them as warn so lint/check still pass while the warnings stay visible. This is a great example of the codebase documenting why a rule is softened rather than silently disabling it.

Prettier's config is tiny — it just loads prettier-plugin-tailwindcss, which auto-sorts Tailwind class names into Tailwind's canonical order. Formatting and linting are kept as separate concerns (Prettier owns whitespace/quotes; ESLint owns code-quality rules), which is why there are distinct format:* and lint* scripts.

Why

projectService: true lets ESLint read TypeScript type information, which is what makes type-aware rules (like catching unawaited promises) possible — plain syntactic linting can't see those.

Tip

When you see comments in eslint.config.js explaining why a rule is a warning not an error, follow that pattern: downgrade-with-a-reason beats silently disabling, so the next dev knows it's a conscious tradeoff, not neglect.

Note

Prettier and ESLint don't overlap here — Prettier handles formatting, ESLint handles code quality. Run format:write and lint:fix for the respective auto-fixes.

Note

Precise rule: @typescript-eslint/no-unused-vars is ["warn", { argsIgnorePattern: "^_" }] — it ignores unused function arguments prefixed with _, but there's no varsIgnorePattern, so an unused local variable still warns even if you underscore-prefix it.

Vitest setup, timezone tests, and Prisma mocking

Vitest is a Vite-native test runner — think 'Jest, but it shares your app's build pipeline,' so ESM, TypeScript, and your path aliases work with almost no extra config. vitest.config.ts sets environment: "node" (these are backend/pure-logic tests, no browser DOM needed) and only picks up src/**/*.test.ts.

The clever bit is the env block. Router-level tests transitively import the app, which pulls in env.js and the Prisma singleton. To stop those imports from throwing when there are no real secrets, the config injects SKIP_ENV_VALIDATION: "true" plus a dummy DATABASE_URL and AUTH_SECRET. Nothing actually connects to a database — the values just satisfy the imports. Coverage uses the v8 provider (@vitest/coverage-v8) scoped to src/lib/**, with a comment noting there's no hard threshold gate yet.

The current test files (timezone.test.ts, rate.test.ts, payment.test.ts, validations.test.ts) are all pure-function tests of src/lib logic — no mocking of Prisma is happening in them today. The vitest-mock-extended package is installed as a devDependency for exactly that purpose (its mockDeep can fabricate a fully-typed fake PrismaClient so you can unit-test a tRPC router without a database), but note for accuracy: it isn't wired into any current test in src — treat it as available tooling, not an existing pattern to copy from.

The timezone tests deserve special mention because they explain the test:tz script. Most assertions are written zone-explicitly (they pass a timezone argument), so they hold no matter what zone the machine is in. But some code paths depend on the host's zone, and — as the test file's own comment notes — mutating process.env.TZ at runtime is a no-op on Windows. So to exercise those paths you must launch the process with the zone already set, which is what cross-env TZ=America/New_York vitest run timezone does. cross-env exists because setting an env var inline (TZ=... cmd) is Unix shell syntax that breaks on Windows PowerShell/cmd — cross-env makes it portable.

Why

The Vitest env block sets SKIP_ENV_VALIDATION + dummy DATABASE_URL/AUTH_SECRET so that merely importing a module which loads env.js doesn't blow up the test — the DB is never actually contacted.

Gotcha

On Windows, setting process.env.TZ inside a test does nothing. A timezone must be fixed before the Node process starts, which is exactly why test:tz uses cross-env TZ=... to launch Vitest rather than setting TZ in code.

Note

vitest-mock-extended is installed to deep-mock PrismaClient, but no current test in src uses it — the existing suite is pure src/lib logic. Don't cite it as an existing example; it's latent tooling.

Why it's built this way

Scoping coverage to src/lib/** (the pure, deterministic logic) reflects a deliberate testing philosophy: test the business rules that are easy to test in isolation and high-value to protect (rates, payments, timezone math), rather than chasing coverage through UI and I/O glue that's better verified end-to-end.

superjson — the transformer that lets Dates survive the wire

Plain JSON.stringify can't represent JavaScript's richer types: a Date becomes a string, a Map/Set becomes {}, BigInt throws, undefined disappears. That's a real problem in this app because Prisma returns lots of Date objects (lesson dates, timestamps) and the data has to travel from the tRPC server to the browser and back. superjson is a drop-in serializer that encodes those types alongside a little metadata, so a Date that leaves the server arrives in the client as an actual Date, not a string you'd have to remember to re-parse.

It's registered in two coordinated places. On the tRPC side, transformer: superjson is passed to initTRPC(...).create(...), so every procedure's input and output is run through superjson automatically. On the React Query / SSR side, query-client.ts wires SuperJSON.serialize/SuperJSON.deserialize into the dehydrate/hydrate steps — that's the mechanism that lets data fetched on the server during SSR be handed to the client without losing its types. Both ends must agree on the transformer, which is why you see it configured in both files.

Analogy

Think of superjson as JSON with a shipping manifest: alongside the packed data it records 'box 3 is a Date, box 5 is a Set,' so the receiving end can rebuild the real objects instead of guessing from strings.

Gotcha

Both the tRPC server and the query client must use the same transformer. If they ever disagree, dates arrive as strings (or worse) and you get subtle 'why is this a string' bugs — so they're configured together on purpose.

Why it's built this way

This app is timezone- and date-heavy, so silently degrading every Date to a string across the network would be a constant source of bugs. superjson buys type fidelity end-to-end for a tiny serialization overhead — a clear win for a data model full of dates and decimals.

The server-only guard and CI

server-only is a one-line safety net for the Next.js server/client boundary. Some modules must never end up in the browser bundle — src/trpc/server.ts is one, because it reads request headers and creates a server-side tRPC caller. Putting import "server-only"; at the very top means: if any client component ever imports this module (directly or through a chain), the build fails with a clear error instead of silently shipping server code — and potentially secrets — to the browser. It's the runtime-boundary cousin of the env split: env.js keeps server values off the client; server-only keeps server modules off the client.

The CI workflow (.github/workflows/ci.yml) is deliberately minimal and mirrors what you can run locally. It triggers on pushes to main and PRs targeting main, with a concurrency group that cancels an in-progress run if you push again (saving minutes). One job on Ubuntu with Node 22: checkout, npm ci (which fires the postinstall → prisma generate so the typecheck has a client), then npm run check (lint + tsc --noEmit), then npm run test. The comments emphasize a key point — none of these steps need secrets: prisma generate never touches the database, and the Vitest config injects dummy env, so CI runs the full gate without any DATABASE_URL or AUTH_SECRET configured. That's a direct payoff of the pure-logic testing approach and the SKIP_ENV_VALIDATION escape hatch.

Next.js build-time font optimizer. Plus_Jakarta_Sans is self-hosted (no runtime Google request, no layout shift) and exposed as the CSS variable class applied to <body>.
next/font/google
A Next.js helper that downloads and self-hosts a Google font at build time, exposing a CSS-variable class — no runtime request to Google. Used here for Plus_Jakarta_Sans.
NEXT_PUBLIC_ prefix
Next.js convention marking an env var as safe to inline into the client bundle. @t3-oss/env-nextjs requires all client vars to use this prefix.
next-themes
Standard Next.js dark-mode library. Installed and read by sonner.tsx, but NOTE: no ThemeProvider is mounted in this repo, so the .dark class is never toggled at runtime yet.
not-found.tsx
The component rendered for a 404 or when code calls notFound(). This repo's version logs the event and offers Go Back / Go Home.
npm run check
The combined quality gate script: npm run lint && tsc --noEmit. CI runs this, so it's the quickest local reproduction of the pipeline (add npm run test for the full gate).
onDelete: Cascade / SetNull
Referential actions on a relation. Cascade deletes child rows when the parent is deleted (lesson→student); SetNull nulls the foreign key but keeps the child (lesson→piece).
onSettled
A mutation callback that runs whether it succeeded or failed — used here to reconcile the cache by invalidating regardless of outcome.
Optimistic update
Editing the cache immediately (before the server responds) so the UI feels instant, with a snapshot kept so onError can roll back if the write fails.
ORM (Object-Relational Mapper)
A library that lets you read/write database rows as typed JS objects and method calls (e.g. db.student.findMany) instead of hand-writing SQL strings. Prisma is this project's ORM.
page.tsx
The special file whose default export renders at its folder's URL. A folder is only a public route if it contains a page.tsx (or route.ts).
Path alias (@/*)
A tsconfig paths mapping (@/* → ./src/*, @prisma/* → ./generated/prisma/*) that replaces long relative imports. Understood by tsc/Next natively and by Vitest via vite-tsconfig-paths.
PaymentMonth
One row per (student, month, year) holding a denormalized expectedAmount snapshot; created lazily on the first payment. Its existence means money has been recorded for that student-month.
PaymentTransaction
An individual payment receipt (amount, method, note, date) hanging off a PaymentMonth. Summing a month's transactions gives receivedAmount.
PrismaAdapter
The NextAuth-Prisma bridge that persists User/Account/Session rows for the database-session/OAuth flows. NOT used in this repo — the JWT + Credentials combination doesn't need it, so it's absent despite the tables existing.
Prisma client singleton
The single shared PrismaClient instance exported from src/server/db.ts, cached on globalThis in development to prevent hot-reload from opening endless connection pools.
prisma db push
Syncs the live database to match schema.prisma by diffing, without creating migration files. This project's chosen workflow (npm run db:push).
prisma generate
The codegen step that reads schema.prisma and writes a typed PrismaClient into node_modules/@prisma/client. Runs automatically via the postinstall script here.
prisma migrate deploy
Replays versioned migration SQL files against a database in order. FORBIDDEN in this project because the migration history has drifted out of sync with the live Neon DB.
Prisma schema (schema.prisma)
The single declarative file describing your datasource, models, fields, relations, enums, and indexes. Everything else (the client types, the DB shape) is derived from it.
Private folder (_components)
An underscore-prefixed folder that the router never treats as a route, used to colocate a page's helper components next to its page.tsx.
Procedure
A single server function in tRPC, either a query (read) or mutation (write), optionally with a validated .input(). The tRPC equivalent of a REST endpoint, but called like a typed function.
protectedProcedure
tRPC middleware that throws TRPCError({ code: "UNAUTHORIZED" }) when ctx.session?.user is absent, and otherwise re-types the context so session.user is non-null in downstream resolvers.
Providers component
The single "use client" file (src/components/providers/index.tsx) that nests all app-wide context (SessionProvider, tRPC/React Query, Zustand store, ErrorBoundary, Toaster) so the root layout can stay a Server Component.
publicProcedure / protectedProcedure
Base building blocks for procedures. Public is open (session may be null). Protected throws UNAUTHORIZED if not logged in and narrows ctx.session.user to non-null for the rest of the procedure.
Query key
The identity of a cached query. In this app tRPC derives it from the procedure path plus input, e.g. ['lesson','getAll', filters]. Same key = same cache entry; a different input object is a different entry.
Radix UI primitive
A headless, accessibility-complete interaction primitive (Dialog, Select, Popover…) that ships behavior and ARIA but no styling. The @radix-ui/react-* packages in package.json.
Rate snapshot
The pattern of copying the student's applicable price onto Lesson.rate at creation and freezing it, so historical lessons keep the price actually charged even after the student's rate changes.
React Server Component (RSC)
A component that runs on the server, ships no JavaScript to the browser, and can be async to await data directly. The default for every file under src/app/ unless it declares "use client".
recharts
Declarative React charting library used for dashboard charts, composed as JSX (<LineChart>, <XAxis>, <Line>).
recommendedTypeChecked
A typescript-eslint rule tier that uses real TypeScript type information (not just syntax) to catch bugs like unawaited promises; requires parserOptions.projectService.
redirect()
A next/navigation function that works by throwing a special control-flow signal, so code written after it does not run. Used in the (root) layout's auth gate and in app/page.tsx.
redirect: false
Option passed to signIn so it resolves or throws an AuthError (catchable, letting the action return a JSON result/toast) instead of performing an HTTP redirect.
Route group
A folder named in parentheses, like (auth) or (root), that groups routes to share a layout WITHOUT adding a URL segment. (root)/dashboard serves /dashboard.
Router
A named group of procedures (and possibly sub-routers). This app has one per feature (student, lesson, payment, ...); the keys you give them become the client namespaces like api.student.
route.ts
An HTTP endpoint handler (not a page). Exports functions named after HTTP verbs (GET, POST, ...) that take a Request and return a Response. Cannot coexist with page.tsx in the same folder.
RSC (React Server Component)
A component that runs on the server, can be async and fetch data directly, and ships no JavaScript to the browser. The default for every file under src/app.
runtimeEnv
The block in env.js that manually maps each schema key to process.env.X — required because Next's edge/client runtimes can't dynamically iterate process.env; each var must be named for the bundler to inline it.
select vs include
select returns only the listed fields of a record; include additionally pulls in related models. You can nest select inside include to fetch only some columns of a relation.
Server Action
A server-side function callable from client code as if local. Arguments and return values must be serializable. Used here for loginAction/registerAction.
Server caller / api (server)
The api from src/trpc/server.ts, built via createCaller. Lets Server Components call procedures directly as async functions (await api.student.getAll()) with no HTTP round-trip. Guarded by import 'server-only'.
Server/client boundary
The seam where a Server Component renders a Client Component. Props crossing it must be serializable (data, Date, Map, Set, `BigInt) — not functions or class instances.
server-only
An npm package whose sole side effect is failing the build if a module importing it is bundled for the browser — guarding server-only modules (e.g. src/trpc/server.ts) from leaking to the client.
session callback
Runs on every auth() call; copies the token's custom fields onto session.user so your app code can read session.user.id / .timezone.
shadcn/ui
A pattern, not a package: component source is copied into src/components/ui/* and owned/edited in-repo. Wraps Radix primitives with this project's Tailwind styling.
skipValidation / SKIP_ENV_VALIDATION
An escape hatch in createEnv that bypasses env schema validation when the SKIP_ENV_VALIDATION variable is set — used for Docker/CI builds and for tests, where real secrets aren't available.
staleTime
How long cached data is considered fresh before React Query is allowed to refetch it on triggers like remount/refocus. Set to 5 minutes here; the library default is 0.
Streaming / Suspense
The mechanism by which the App Router sends the shell + loading fallback immediately, then streams in real content when async Server Components finish. Surfaced via loading.tsx.
summarizeOutstanding
All-time totals helper that buckets expected and received per student-month and clamps remaining per bucket before summing, so an overpaid month can't offset an underpaid one.
superjson
The transformer configured on both server and client that serializes rich types (Date, Map, BigInt, undefined) which plain JSON can't. It's why lesson.date arrives on the client as a real Date. Must match on both ends.
suppressHydrationWarning
A React attribute (used on <html> in the root layout) that silences the harmless server/client mismatch warning for that single element, typically when a class or theme is set before hydration.
@t3-oss/env-nextjs
A small T3-stack library that wraps Zod to validate environment variables at build/boot, enforce a server/client split, and expose them as a typed env object. Configured in src/env.js.
T3 Stack
The opinionated starter (create-t3-app) that wires Next.js, TypeScript, tRPC, Prisma, NextAuth, Tailwind, and Zod together type-safely. This project was generated from it.
Tailwind CSS v4 (config-light)
The version used here. Configuration lives in CSS (@import "tailwindcss" + @theme tokens in globals.css) instead of a tailwind.config.js file, which this repo does not have.
tailwind-merge (twMerge)
Understands Tailwind class groups and drops earlier conflicting utilities (px-2 vs px-4), so the last class for a property wins regardless of stylesheet order.
@tanstack/react-table
Headless table library (sorting/filtering/pagination/row models, no markup). Built via useReactTable; rendered by the shared DataTable using flexRender.
test:tz
npm script cross-env TZ=America/New_York vitest run timezone that runs the timezone suite under a forced host zone to catch code that leaks the machine's timezone (needed because process.env.TZ is a runtime no-op on Windows).
@theme block
A CSS block in globals.css where design tokens are declared. Each token (e.g. --color-primary, --radius-4xl) becomes both a CSS variable and a Tailwind utility.
tRPC React client (`@/trpc/react`)
A createTRPCReact api exposing React Query hooks (useQuery, etc.) that call procedures over HTTP (/api/trpc) with SuperJSON batching. Used inside Client Components.
tRPC server caller (`@/trpc/server`)
A createCaller-based api object that invokes tRPC procedures as in-process function calls with no HTTP. Marked import "server-only"; used inside Server Components.
tw-animate-css
Small animation-utility library imported in globals.css, providing the animate-in / fade-in-0 / zoom-in-95 classes used with Radix's data-[state] attributes for enter/exit transitions.
Type-safe env
The pattern of validating process.env against a schema once at the boundary so the rest of the app reads guaranteed-present, correctly-typed values (e.g. env.DATABASE_URL) instead of string | undefined.
@@unique (composite unique)
A uniqueness constraint across several columns (e.g. @@unique([studentId, month, year])). Prevents duplicates and provides the key that upsert targets (named studentId_month_year).
upsert
A single operation that updates the row matching a unique key if it exists, or creates it otherwise — used here on the studentId_month_year composite key for reports and payment months.
"use client"
A directive on line 1 of a file that marks it (and its client-imported dependencies) as browser-side. It's a coarse per-file boundary, not a per-component switch.
useMutation
tRPC/React Query hook for a write. Returns mutate/mutateAsync plus isPending and onSuccess/onError/onSettled callbacks. Does not auto-update reads — you invalidate.
"use server"
A directive that marks a file's exports (or a single function) as Server Actions — server-run functions that Client Components can call directly, with Next.js handling the RPC transport.
useUtils()
tRPC's typed handle to the underlying React Query cache. utils.<router>.<proc>.invalidate()/.setData()/.prefetch() targets a query by its typed path instead of a stringly-typed key.
vaul
Headless drawer/bottom-sheet primitive wrapped by src/components/ui/drawer.tsx, with direction-aware styling via data-vaul-drawer-direction selectors.
Vitest
A Vite-native test runner (Jest-like API) used here in Node environment for pure src/lib logic tests; shares the app's build pipeline so TS and path aliases work with minimal config.
vitest-mock-extended
A library whose mockDeep can produce a fully-typed fake PrismaClient for unit-testing DB-touching code without a database. Installed here but not yet used by any test in src.
vite-tsconfig-paths
A Vite/Vitest plugin that reads tsconfig.json's paths so test imports resolve the same aliases the app uses, keeping one source of truth.
zodResolver
Adapter (@hookform/resolvers/zod) that lets React Hook Form validate against a Zod schema, so one schema drives both TypeScript types (z.infer) and runtime validation.
Zustand
A minimal global state library. create() returns a hook (useUserStore) with no provider needed; the persist middleware mirrors state to localStorage.
